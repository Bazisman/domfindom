# INSTRUCTIONS

> Обновлено: 2026-04-18
> Назначение: рабочие правила для ИИ-ассистента внутри этого репозитория.

## Общие принципы

ИИ в этом проекте должен:
- сначала понять текущее состояние кода и продукта;
- не спорить с уже принятыми решениями без причины;
- обновлять документацию, когда меняется реальное поведение системы;
- относиться к кодировке как к зоне повышенного риска.

## Как действовать перед началом задачи

1. Прочитать `TASKS.md`.
2. Прочитать `docs/PROJECT_STATE.md`.
3. Прочитать `docs/DECISIONS.md`.
4. Прочитать `docs/WORKING_MEMORY.md`.
5. Прочитать последние записи в `docs/WORK_LOG.md`.
6. При необходимости посмотреть `docs/PROJECT_HISTORY.md`.
7. Только после этого идти в конкретный код.

## Приоритеты по коду

Основной активный продукт сейчас находится здесь:
- `backend/`
- `frontend/`
- `backend/auth/`

Desktop-часть не удалена, но не должна задавать направление новых продуктовых решений.

## Правила редактирования

- Вносить минимально необходимые изменения.
- Не переписывать без нужды большие рабочие блоки.
- Если меняется пользовательский сценарий, синхронно обновлять документацию.
- Если меняются русские строки, письма или markdown-файлы, обязательно помнить про кодировку.

## Если пользователь пишет `запротоколируй`

Нужно:
- обновить релевантные опорные документы;
- дополнить `docs/WORK_LOG.md`;
- зафиксировать, что сделано, что проверено и какой следующий шаг.

`docs/archive/` использовать только если реально нужен отдельный архивный отчет.

## Правила по кодировке

### Обязательный стандарт

- все текстовые файлы: `UTF-8` без BOM;
- не использовать случайные перекодировки между `cp1251`, `utf8`, `latin1` и подобными схемами;
- при любом сомнении предпочитать явную запись файла в UTF-8.

### Проверка

```bash
python tools/check_encoding.py --root .
```

### Для PowerShell

Если нужно явно писать файл, безопаснее использовать .NET-способ без BOM:

```powershell
[System.IO.File]::WriteAllText(
  $path,
  $content,
  (New-Object System.Text.UTF8Encoding($false))
)
```

## Важные продуктовые инварианты

- `display_name` должен иметь приоритет над email в UI.
- Семейный и личный режимы не смешиваются без явного решения.
- Семья — это общий обзор и роли, а не наивное слияние всех личных данных в одно хранилище.
- Опасные действия должны быть понятными и подтверждаемыми.
- Email-flow нужно проверять не только в API, но и в письмах и на экране логина.

## Что проверять после заметных изменений

Минимальный набор зависит от задачи, но обычно это:
- `python tools/check_encoding.py --root .`
- `npm run build` в `frontend/`, если менялся фронтенд
- релевантные тесты в `tests/`
- если менялся production-flow: health endpoint после деплоя
