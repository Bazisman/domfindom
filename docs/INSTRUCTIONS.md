# INSTRUCTIONS

> Обновлено: 2026-04-18
> Назначение: рабочие правила для ИИ-ассистента внутри этого репозитория.

## Общие принципы

ИИ в этом проекте должен:
- сначала понять текущее состояние кода и продукта;
- не спорить с уже принятыми решениями без причины;
- обновлять документацию, когда меняется реальное поведение системы;
- относиться к кодировке как к зоне повышенного риска.

## Как действовать перед началом задачи

1. Прочитать `TASKS.md`.
2. Прочитать `docs/PROJECT_STATE.md`.
3. Прочитать `docs/DECISIONS.md`.
4. Прочитать `docs/WORKING_MEMORY.md`.
5. Прочитать последние записи в `docs/WORK_LOG.md`.
6. При необходимости посмотреть `docs/PROJECT_HISTORY.md`.
7. Только после этого идти в конкретный код.

### Если пользователь пишет `начинаем`

Это специальный короткий маркер старта новой сессии.

Он означает, что ассистент обязан:
- выполнить полный стартовый протокол чтения документов из списка выше;
- восстановить из документов не только контекст проекта, но и сам способ работы:
  - какие правила обязательны;
  - как здесь правильно делать правки;
  - как безопасно выполнять git и deploy;
  - какие антиошибки уже накоплены и не должны повторяться;
- коротко пересказать, что понял по текущему состоянию проекта и по правилам работы;
- только потом переходить к реализации.

Важно:
- `начинаем` не означает автоматический `git commit`;
- `начинаем` не означает автоматический `git push`;
- `начинаем` не означает автоматический деплой.

Отдельная пользовательская договоренность для этого проекта:
- если задача явно выглядит как рабочий продуктовый фикс и пользователь не просит оставить изменения только локально, ассистенту нужно по умолчанию доводить работу до `git`, `push` и production-deploy;
- если пользователь хочет только локальные изменения без выкладки, это должно быть сказано явно.

Для короткой памятки см. `docs/CHAT_START.md`.

## Приоритеты по коду

Основной активный продукт сейчас находится здесь:
- `backend/`

Desktop-часть не удалена, но не должна задавать направление новых продуктовых решений.

## Правила редактирования

- Вносить минимально необходимые изменения.
- Не переписывать без нужды большие рабочие блоки.
- Если меняется пользовательский сценарий, синхронно обновлять документацию.
- Если меняются русские строки, письма или markdown-файлы, обязательно помнить про кодировку.
- Если `apply_patch` не сел, нельзя продолжать тупо повторять тот же патч: нужно сначала дочитать точный актуальный фрагмент файла и только потом вносить адресную правку по реальному состоянию кода.
- Перед `git add` и `git commit` учитывать, что в этом проекте возможен `index.lock`/permission issue: в таком случае нужно сразу переходить к точечному staged-flow с повышенными правами и добавлять только явный список нужных файлов, без `git add -A`.
- При деплое нельзя параллелить шаги `upload archive` и `extract on host`: сначала дождаться успешной загрузки архива на сервер, и только потом отдельно выполнять распаковку и restart.
- Если фронтенд уже сломан визуально или текстом, нельзя сначала продолжать улучшения поверх поломки: сначала вернуть последний подтвержденно чистый файл или коммит, проверить кодировку и сборку, и только потом снова идти в UX-правки.
- Для UI-карточек и других плотных блоков нельзя дублировать один и тот же смысл в соседних секциях: если данные уже есть в `Текущем балансе`, в карточках прогноза нужно оставлять только собственный смысл блока.
- При точечных фронтенд-правках безопаснее переносить только нужную механику из прежней рабочей версии, а не смешивать в одном изменении анимацию, новые тексты, новую структуру и побочные улучшения.

## Если пользователь пишет `запротоколируй`

Нужно:
- обновить релевантные опорные документы;
- дополнить `docs/WORK_LOG.md`;
- зафиксировать, что сделано, что проверено и какой следующий шаг.

Правило стиля для `docs/WORK_LOG.md`:
- писать коротко и по смыслу результата в проекте, а не по списку собственных действий ассистента;
- хорошая запись отвечает на вопрос "что изменилось в продукте/логике/процессе", а не "что я делал руками";
- prefer строки вида `На странице X ...`, `Для сценария Y ...`, `В проекте закрепили ...`;
- избегать строк в стиле `добавил`, `сделал`, `выполнил`, если это не запись про проверку или выкладку;
- технические проверки и деплой выносить отдельными строками ниже, не смешивая их с продуктовым смыслом.

`docs/archive/` использовать только если реально нужен отдельный архивный отчет.

## Правила по кодировке

### Обязательный стандарт

- все текстовые файлы: `UTF-8` без BOM;
- не использовать случайные перекодировки между `cp1251`, `utf8`, `latin1` и подобными схемами;
- при любом сомнении предпочитать явную запись файла в UTF-8.

### Проверка

```bash
python tools/check_encoding.py --root .
```

### Для PowerShell

Если нужно явно писать файл, безопаснее использовать .NET-способ без BOM:

```powershell
[System.IO.File]::WriteAllText(
  $path,
  $content,
  (New-Object System.Text.UTF8Encoding($false))
)
```

## Важные продуктовые инварианты

- `display_name` должен иметь приоритет над email в UI.
- Семейный и личный режимы не смешиваются без явного решения.
- Семья — это общий обзор и роли, а не наивное слияние всех личных данных в одно хранилище.
- Опасные действия должны быть понятными и подтверждаемыми.
- Email-flow нужно проверять не только в API, но и в письмах и на экране логина.

## Что проверять после заметных изменений

Минимальный набор зависит от задачи, но обычно это:
- `python tools/check_encoding.py --root .`
- `npm run build` в `frontend/`, если менялся фронтенд
- релевантные тесты в `tests/`
- если менялся production-flow: health endpoint после деплоя
- если фикс связан с русским текстом или фронтенд-карточками: дополнительно убедиться, что в собранной версии не появилось mojibake и что упрощение UI не дублирует соседний блок по смыслу

## Короткий протокол безопасного деплоя

1. Локально прогнать проверки и сборку.
2. Сделать commit и push только нужных файлов.
3. На production отдельно выполнить `git pull --ff-only` для backend.
4. Отдельным шагом загрузить frontend-архив на хост.
5. Только после завершения загрузки отдельно распаковать архив в web-root.
6. Выполнить restart.
7. Проверить health endpoint.
