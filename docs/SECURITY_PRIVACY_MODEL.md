# Модель безопасности и приватности данных

> Обновлено: 2026-04-25
> Статус: целевая модель и поэтапный план. Не все уровни уже реализованы.

## 1. Главный пользовательский вопрос

Пользователь спрашивает:

> Разработчик сможет читать мои деньги?

Правильный ответ должен быть честным:
- сейчас доступ к данным защищен учетной записью, сессиями, ролями, per-user SQLite-файлами и production-дисциплиной;
- целевая модель должна сделать так, чтобы финансовые данные нельзя было просто открыть из дампа базы;
- максимальная приватность требует пользовательского ключа, без которого даже разработчик не сможет штатно расшифровать данные;
- доступ поддержки должен выдаваться пользователем явно, временно и с аудитом.

## 2. Что уже есть

В проекте уже реализованы:
- регистрация и вход;
- `HttpOnly` cookie-сессии;
- CSRF-защита для приватных write-запросов;
- email verification;
- password reset;
- account deletion через email-подтверждение;
- журнал auth-событий;
- активные сессии;
- per-user finance DB routing;
- семейные роли `owner`, `member`, `viewer`;
- owner-only действия для приглашений и аудита категорий;
- production-требование строгого `FINANCE_APP_SESSION_SECRET`.

Пароли сейчас хешируются через `scrypt`. При следующей security-модернизации стоит отдельно рассмотреть переход на `Argon2id` как предпочтительный современный вариант для password hashing.

## 3. Чего сейчас нет

Сейчас не реализовано:
- шифрование финансовых полей перед записью в БД;
- отдельное хранилище ключей;
- пользовательский ключ, недоступный серверу;
- криптографический grant для временного доступа поддержки;
- ротация ключей шифрования данных;
- privacy-аудит чтения финансовых данных.

Это значит:
- доступ к production-файлам или базе должен считаться чувствительным административным доступом;
- пользователям нельзя обещать, что разработчик технически не способен прочитать данные, пока не внедрен user-held key.

## 4. Целевая модель защиты

Цель:
- пользовательские финансовые данные хранятся зашифрованными;
- ключи отделены от самой базы;
- штатный доступ разработчика к данным невозможен без явного действия пользователя;
- любой временный доступ поддержки фиксируется и может быть отозван.

Целевая цепочка:

```text
Финансовые данные
  -> шифруются user_data_key

user_data_key
  -> защищен ключом пользователя или envelope-механизмом

Поддержка
  -> получает доступ только через support grant,
     созданный на стороне пользователя
```

## 5. Уровни реализации

### Уровень 1. Application-level encryption

Первый практичный уровень после или вместе с PostgreSQL-подготовкой.

Как работает:
- чувствительные поля шифруются приложением перед записью в БД;
- PostgreSQL хранит ciphertext, nonce и metadata версии ключа;
- ключи хранятся отдельно от БД;
- утечка дампа PostgreSQL не раскрывает данные напрямую.

Что защищает:
- от чтения дампа БД;
- от случайного просмотра raw database;
- от части эксплуатационных рисков.

Чего не защищает полностью:
- от администратора приложения, у которого есть доступ и к БД, и к server-side ключу;
- от вредоносного кода на сервере.

Этот уровень полезен, но он еще не означает “даже разработчик не может прочитать”.

### Уровень 2. User-held key mode

Настоящий приватный режим.

Как работает:
- пользователь задает отдельный секрет для финансовых данных;
- data key расшифровывается только после действия пользователя;
- сервер не хранит секрет в открываемом виде;
- без пользовательского секрета разработчик не может штатно расшифровать данные.

Важный компромисс:
- если пользователь потеряет секрет, восстановление данных может быть невозможно;
- фоновые расчеты и семейные агрегаты становятся сложнее;
- часть вычислений может потребоваться переносить в браузер или выполнять только во время активной сессии пользователя.

### Уровень 3. Support access grant

Сценарий “пользователь временно разрешил поддержку”.

Как работает:
1. У поддержки есть публичный и приватный ключ.
2. Пользователь нажимает `Разрешить доступ поддержке`.
3. Браузер пользователя локально получает доступ к `user_data_key`.
4. Браузер шифрует временный grant публичным ключом поддержки.
5. Сервер сохраняет grant и audit record.
6. Разработчик открывает grant своим приватным ключом.
7. Доступ ограничен временем, пользователем и причиной.
8. Пользователь может отозвать grant.

Email в этой схеме нужен для подтверждения и уведомлений, но не для пересылки главного ключа.

## 6. Что не стоит делать

Не стоит:
- отправлять главный ключ пользователю на почту;
- просить пользователя пересылать секрет разработчику в чат;
- считать email-токен полноценным ключом шифрования;
- хранить ключ рядом с зашифрованными данными без защиты;
- внедрять самописную криптографию;
- обещать “абсолютную защиту”.

Правильная формулировка:

> Доступ поддержки возможен только после явного разрешения пользователя и фиксируется в журнале.

После внедрения user-held key можно будет усилить:

> Без пользовательского разрешения разработчик не может штатно расшифровать финансовые данные.

## 7. Что шифровать

Кандидаты на шифрование:
- суммы операций;
- комментарии операций;
- названия счетов;
- названия пользовательских категорий;
- остатки счетов;
- backup-слоты;
- семейные финансовые агрегаты, если они сохраняются.

Технические поля, которые могут остаться открытыми ради индексов и фильтрации:
- `user_id`;
- `family_id`;
- тип записи;
- дата или период;
- status;
- timestamps;
- key/version metadata.

Каждое открытое поле должно быть осознанным компромиссом, а не случайностью.

## 8. Влияние на семейный режим

Семейный режим сложнее личного, потому что данные нескольких пользователей агрегируются.

Открытые вопросы:
- должен ли владелец семьи иметь доступ к расшифровке операций участников;
- можно ли считать семейный forecast без активных ключей всех участников;
- где расшифровывать семейные данные: на сервере во время активной сессии или в браузере;
- как отозвать семейный доступ при выходе участника из семьи;
- как работают category bindings, если названия категорий зашифрованы.

До ответов на эти вопросы нельзя внедрять строгий user-held key сразу для всего семейного режима.

## 9. Связь с PostgreSQL

Переход на PostgreSQL и privacy-модель нужно проектировать вместе, но внедрять по очереди.

Безопасный порядок:
1. Инвентаризация SQLite.
2. PostgreSQL ER-модель с учетом будущих ключей.
3. Миграция на PostgreSQL без изменения смысла данных.
4. Application-level encryption.
5. User-held key для личного режима.
6. Support access grant.
7. Расширение строгой приватности на семейные сценарии.

## 10. Первые технические решения

Предварительно:
- парольные хеши: перейти к `Argon2id` при отдельной задаче hardening;
- шифрование данных: использовать только стандартные AEAD-алгоритмы (`AES-GCM` или `ChaCha20-Poly1305`);
- ключи: per-user data key, отдельная metadata версии ключа;
- support grant: public-key encryption через проверенную библиотеку, без самописных алгоритмов;
- аудит: отдельный журнал выдачи, использования и отзыва доступа поддержки.

Финальный выбор библиотек делать отдельной задачей после проверки совместимости с production Python/hosting.

## 11. Первый безопасный следующий шаг

Зафиксировать модель в документации и не менять runtime-код до отдельного технического проекта.

Практический следующий шаг:
- сделать read-only инвентаризацию текущих БД;
- подготовить PostgreSQL ER-модель с местами под encryption metadata;
- отдельно решить, какой уровень privacy включать первым.
