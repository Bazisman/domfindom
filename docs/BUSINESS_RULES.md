# Бизнес-правила

> Обновлено: 2026-04-25
> Назначение: сухо зафиксировать расчетные правила, формулы и инварианты продукта без длинного описания экранов.

## 1. Общие принципы

- Один показатель должен иметь один смысл.
- Нельзя смешивать `текущий баланс` и `результат месяца`.
- Факт и прогноз должны быть разделены.
- Личный и семейный контексты должны считаться по разным данным.
- Planned-операции и бюджетные правила — это разные слои логики.

## 2. Базовые формулы

### 2.1. Текущий баланс

`current_balance = opening_balance + executed_income_to_date - executed_expense_to_date`

Где:
- `opening_balance` — входящий остаток прошлых периодов;
- `executed_income_to_date` — уже полученные доходы текущего месяца;
- `executed_expense_to_date` — уже исполненные расходы текущего месяца.

### 2.2. Доход месяца

`month_income = executed_income_to_date + planned_income_until_month_end`

### 2.3. Расход месяца

`month_expense = executed_expense_to_date + planned_expense_until_month_end + budget_future_expense`

### 2.4. Результат месяца

`month_result = month_income - month_expense`

## 3. Правила по бюджетам

## 3.1. Общий инвариант

Один и тот же бюджет может иметь разную прикладную семантику в разных экранах:
- в `forecast`;
- в `Статусе бюджетов`;
- в предупреждениях при вводе расхода.

Нельзя переносить формулу из одного экрана в другой без проверки смысла.

### 3.2. Daily-бюджет в forecast

Обозначения:
- `a` — дневной лимит;
- `b` — число дней в месяце;
- `z` — уже потрачено по категории на текущую дату;
- `days_left` — оставшиеся дни до конца месяца, включая сегодня;
- `d = a * days_left`;
- `v = a * b`;
- `g = v - (z + d)`.

Правила:
- `budget_future_expense_for_category = d`;
- `z` не вычитается повторно в прогнозе;
- `v` не должен целиком уходить в прогноз поверх уже случившегося факта.

Уточнение по текущему дню:
- если сегодня по daily-категории уже есть фактический расход, `days_left` для forecast считается с завтрашнего дня;
- если сегодня фактического расхода еще нет, `days_left` включает сегодня.

Формально:
- `days_to_reserve = days_after_today`, если `spent_today > 0`;
- `days_to_reserve = days_including_today`, если `spent_today == 0`;
- `budget_future_expense_for_category = a * days_to_reserve`.

### 3.3. Monthly-бюджет в forecast

Базовое правило:

`budget_future_expense_for_category = max(monthly_limit - spent, 0)`

Дополнительный инвариант:
- если продуктово принято считать planned-расходы поверх бюджета, они не должны уменьшать этот бюджетный остаток внутри категории.

### 3.4. Planned и бюджет не взаимозаменяемы

Инвариант:
- `planned_expense` — это слой точных будущих операций;
- `budget_future_expense` — это слой бюджетного резерва или прогнозного поведения;
- одно не должно автоматически поглощать другое, если это не зафиксировано как отдельное правило продукта.

## 4. Правила `Статуса бюджетов`

### 4.1. Monthly / Yearly

Для `monthly` и `yearly` в статусе бюджета:

- `budget_amount = limit_for_period`
- `spent = executed_expense_to_date`
- `remaining = budget_amount - spent`
- `percent = spent / budget_amount`

### 4.2. Daily

Для `daily` в статусе бюджета:

- `z` — уже потрачено по категории;
- `days_left` — оставшиеся дни месяца, включая сегодня;
- `d = daily_amount * days_left`;
- `budget_amount = z + d`;
- `remaining = d`;
- `percent = z / (z + d)`.

Смысл:
- карточка показывает ожидаемый итог категории к концу месяца;
- `remaining` показывает будущий расход на остаток месяца.

### 4.3. Что нельзя делать для daily в статусе бюджета

Нельзя использовать как `budget_amount`:
- полный месячный лимит `daily_amount * days_in_month`;
- лимит только по прошедшим дням `daily_amount * elapsed_days`.

Причина:
- оба варианта не совпадают с пользовательским ожиданием для этого экрана.

## 5. Личный и семейный контекст

### 5.1. Личный режим

Все агрегаты строятся только по данным текущего пользователя.

### 5.2. Семейный режим

Все агрегаты строятся как сумма по участникам семьи.

Инварианты:
- семейный факт = сумма фактических операций участников;
- семейный planned = сумма planned-операций участников;
- семейные лимиты = лимиты всех участников семьи по общему смыслу категории;
- семейный budget status = расчет от семейного факта и семейных лимитов, а не от факта и локальных лимитов текущего viewer.

Если участник семьи не создавал локальный бюджет, но владелец или другой участник создал семейно значимый лимит, этот участник все равно должен видеть семейный статус бюджета.

Категория, оставленная личной в аудите:
- остается в денежных суммах семьи как факт расхода или дохода;
- исключается из семейных лимитов и семейного бюджетного прогноза;
- остается доступной для личных лимитов пользователя.

### 5.3. Антиправило для семейного daily status

Запрещено:
- брать personal `budget_amount` из `daily`-карточки;
- заменять только `spent` на семейный.

Причина:
- тогда `budget_amount` перестает соответствовать семейному факту и карточка становится логически неверной.

## 6. Инварианты интерфейса

- Если в UI используется термин `текущий баланс`, он не должен означать `результат месяца`.
- Если используется термин `остаток на конец месяца`, он не должен означать деньги на руках прямо сейчас.
- В соседних карточках нельзя дублировать один и тот же смысл под разными заголовками.
- Обычный участник семьи не должен видеть owner-only действия: приглашения, управление ролями и аудит категорий.
- Участник семьи должен видеть семейные агрегаты и семейные лимиты без необходимости иметь собственные локальные лимиты.
- Если формула расчета поменялась, это должно быть синхронизировано с документацией.
