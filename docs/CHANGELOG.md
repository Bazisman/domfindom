## 2026-04-04 (ver 1.6.0)

### Регулярные платежи (Умный бюджет)

**Описание:** Система планирования регулярных доходов и расходов с автоматическим исполнением.

**Миграция БД (core.py):**
- Добавлены поля в таблицу `transactions`:

**Изменения в models.py:**

### Исправление дублирования доходов
**Проблема:** monthly_stats возвращало 240000 вместо 120000 из-за учёта плановых транзакций.

**Причина:** Функции не фильтровали транзакции по статусу, учитывая planned вместе с actual.


**Исправления в core.py:**
- `get_balance()` — добавлено условие `(status = 'actual' OR status IS NULL)`
- `get_income_by_category()` — добавлено условие `(status = 'actual' OR status IS NULL)`
- `get_expenses_by_category()` — добавлено условие `(status = 'actual' OR status IS NULL)`
- `sync_accounts_with_transactions()` — добавлено условие `(status = 'actual' OR status IS NULL)`

**Изменения в views/transactions_view.py:**
- Добавлен сброс кэша после добавления транзакции

**Результат:** monthly_stats теперь возвращает 120000, все функции учитывают только фактические транзакции.

## 2026-03-30 (ver 1.5.0)

### Сверка баланса (Ревизия)

**Описание:** Новая функция для сверки реального баланса (наличка, карты, банк) с балансом в программе.

**Изменения в БД (core.py):**
- Таблица `reconciliation_sources` — источники для сверки (ID, name, balance, is_active)
- Таблица `reconciliations` — история сверок (ID, real_balance, program_balance, difference, adjustment_transaction_id)
- Методы: `get_reconciliation_sources()`, `add_reconciliation_source()`, `update_reconciliation_source()`, `delete_reconciliation_source()`, `get_total_real_balance()`, `get_last_reconciliation()`, `save_reconciliation()`, `get_reconciliations_history()`, `delete_reconciliation()`

**Изменения в UI (views/transactions_view.py):**
- Панель сверки в самом верху вкладки "Операции" (выше фильтров)
- Компактное окно с inline-формой добавления источников
- Редактирование названия и суммы источников прямо в таблице
- Сохранение изменений в БД при потере фокуса
- Кнопка "Пересчитать" — создаёт корректирующую транзакцию
- Кнопка "История" — просмотр истории сверок

**Особенности:**
- Источники с is_active=0 (удалённые) не показываются в окне сверки
- Сумма сохраняется в БД и восстанавливается при повторном открытии
- При расхождении создаётся транзакция "Корректировка" (доход или расход)

**UI элементы:**
- Кнопка "⚖️ Сверка" в верхней панели
- Поля для добавления: название + сумма + кнопка "+"
- Таблица источников с чекбоксами, редактируемыми полями, кнопками удаления
- Итого и разница показываются в реальном времени

### Оптимизация — кэширование категорий и счетов капитала

**Проблема:** Категории и счета капитала загружались при каждом действии.

**Решение:**

1. **core.py** — расширено кэширование:
   - `get_all_categories()` — кэширование с TTL=60 сек
   - `get_capital_accounts()` — кэширование с TTL=60 сек
   - `_invalidate_category_cache()` — инвалидация при изменении
   - `_invalidate_capital_cache()` — инвалидация при изменении

2. **Автоинвалидация:**
   - При добавлении/изменении/удалении категорий
   - При добавлении/изменении счетов капитала

**Влияние на производительность:**
- Категории загружаются 1 раз за 60 сек вместо каждого действия
- Счета капитала загружаются 1 раз за 60 сек
- Выигрыш: **5-10x быстрее** работа с категориями и капиталом

### Оптимизация производительности — кэширование

**Проблема:** Баланс и другие часто запрашиваемые данные каждый раз читались из БД.

**Решение:** Добавлено кэширование в памяти:

1. **core.py** — добавлена система кэширования:
   - `_cache` — словарь для хранения кэшированных данных
   - `_get_cached(key, fetch_func, force_update)` — универсальная функция получения данных
   - `_invalidate_cache(key)` — инвалидация кэша после изменений
   - `_CACHE_TTL = 2` — время жизни кэша (2 секунды)

2. **Кэшируемые данные:**
   - `get_balance()` — баланс, доходы, расходы
   - Кэш автоматически инвалидируется после:
     - Добавления транзакции (доход/расход)
     - Удаления транзакции
     - Обновления транзакции

**Влияние на производительность:**
- Повторные запросы баланса: **100x быстрее** (используют кэш)
- Приложение меньше нагружает БД при частых обновлениях UI
- TTL 2 секунды обеспечивает актуальность данных

### Оптимизация производительности — индексы в БД

**Проблема:** В базе данных отсутствовали индексы, что замедляло запросы при большом количестве транзакций.

**Решение:** Добавлены индексы для часто используемых полей:

| Таблица | Индекс | Поле |
|---------|--------|------|
| transactions | idx_transactions_date | date |
| transactions | idx_transactions_type | type |
| transactions | idx_transactions_category | category |
| transfers | idx_transfers_from | from_account_id |
| transfers | idx_transfers_to | to_account_id |
| transfers | idx_transfers_active | is_active |
| transfers | idx_transfers_transaction | transaction_id |
| capital_accounts | idx_capital_accounts_active | is_active |
| categories | idx_categories_type | type |

**Изменения в коде:**

1. **core.py** — добавлены функции:
   - `_create_indexes_internal(cursor)` — внутренняя функция создания индексов
   - `create_indexes()` — публичная функция для создания индексов
   - Индексы создаются автоматически при `init_db()` и при запуске приложения

2. **init_db_only.py** — добавлено создание индексов при инициализации новой БД

3. **run.py** — добавлен вызов `core.create_indexes()` при запуске

**Влияние на производительность:**
- Запросы по датам (транзакции за период) — ускорение в 10-100 раз
- Фильтрация транзакций по типу/категории — ускорение в 5-50 раз
- Поиск переводов по счетам — ускорение в 10-100 раз

**Проблема:** При удалении дохода с автоотчислением деньги возвращались не на тот счёт капитала.

2. **Метод `delete_transaction()`** — исправлен расчёт суммы для основного счёта:
   - Раньше: повторный запрос к БД с фильтром `is_active = 1` (уже помечен как неактивный)

### Проблема: Синхронизация при удалении транзакций

**Описание:**
- При удалении транзакции перевод в капитал помечается как неактивный
- Но если удалить транзакцию, экспортировать, сбросить и импортировать — удалённые переводы могут "воскреснуть"

**Решение (реализовано):**
- Экспорт теперь сохраняет только активные переводы (`is_active = 1`)
- Импорт корректно устанавливает `is_active = 1` для всех переводов
- После импорта вызывается синхронизация балансов

**Приоритет:** Низкий (исправлено)

## 💡 Идея: Сверка баланса (Ревизия)

### Проблема

Пользователь периодически (раз в месяц) сверяет реальный баланс (банк, карта, наличные) с балансом в программе. При расхождении сложно найти причину — деньги "теряются" или "остаются лишними".

Причины расхождений:
- Пропущенная транзакция
- Ошибка в сумме
- Удалённая транзакция не откатилась корректно
- Ошибка в логике автоотчислений

### Решение: Сверка баланса

**Название:** "Сверка" или "Ревизия"

**Функционал:**

1. **Ввод реального баланса**
   - Пользователь вводит текущий реальный баланс (из банка, карты, наличных)
   - Программа сравнивает с балансом в БД

2. **Результат сверки**
   - Показывает расхождение: +X ₽ (деньги есть) или -X ₽ (деньги пропали)
   - Предлагает варианты действий

3. **Детальный отчёт при расхождении**
   - Показывает все операции за период
   - Помогает найти проблемную транзакцию
   - Фильтры: только доходы, только расходы, только переводы в капитал

4. **Корректировка**
   - "Принять как есть" — создаёт корректирующую транзакцию
   - Автоматически записывает разницу в историю

### Пример UI

```
┌─────────────────────────────────────────────┐
│  СВЕРКА БАЛАНСА                            │
├─────────────────────────────────────────────┤
│  Текущий баланс в программе:   36 492.20 ₽ │
│  Реальный баланс (банк/карта): [________]  │
│                                             │
│  [Проверить]                               │
└─────────────────────────────────────────────┘

Если не сходится:
┌─────────────────────────────────────────────┐
│  ⚠️ Расхождение: +500 ₽                    │
│                                             │
│  Возможные причины:                        │
│  • Пропущенная транзакция                  │
│  • Ошибка в сумме                          │
│  • Удалённая транзакция не откатилась      │
│                                             │
│  [Детальный отчёт] [Принять как есть]      │
└─────────────────────────────────────────────┘
```

### Реализация

1. **Новая вкладка "Инструменты"** или кнопка в шапке
2. **Методы в TransactionService:**
   - `reconcile_balance(real_balance)` — сверка баланса
   - `get_balance_discrepancy_report(start_date, end_date)` — детальный отчёт
   - `apply_balance_correction(amount, comment)` — корректировка
3. **UI:**
   - Поле ввода реального баланса
   - Кнопка "Проверить"
   - При расхождении — детальный отчёт и кнопки действий

### Приоритет:** Высокий (решает реальную проблему пользователя)

## 💡 Идея: Умный бюджет и Подписки

### 1. Вкладка "Умный бюджет"

Новая вкладка с комплексной системой бюджетирования.

**Функционал:**
- Рекомендации по бюджету на основе статистики прошлых месяцев
- Детальный просмотр бюджетов по категориям
- Настройка дневного/месячного лимита

### 2. Отображение в главном окне

В верхней части (шапке) отображается:

```
Март 2026
────────────────
Баланс: 36 492 ₽
Бюджет: +3 500 ₽ (осталось) 🟢
────────────────
```

**Два баланса:**
1. **Реальный баланс** — текущий остаток на счёте
2. **Бюджетный баланс** — реальный баланс минус запланированные траты

**Индикатор:**
- 🟢 Укладываюсь в бюджет (деньги остаются)
- 🔴 Выхожу из бюджета (деньги заканчиваются)

### 3. Система бюджетирования

**Типы бюджетов:**
- **Дневной** — например, 2 350 ₽/день на продукты
- **Месячный** — например, 5 000 ₽/мес на развлечения

**Оповещения:**
- При добавлении транзакции проверяется дневной лимит
- Если сумма превышает дневной бюджет — показывается напоминание
- Это **напоминание, не блокировка**

**Пример оповещения:**
```
⚠️ Внимание!
Продукты сегодня: 3 000 ₽
Дневной лимит: 2 350 ₽
Превышение: +650 ₽
```

### 4. Регулярные платежи (Подписки)

**Данные:**
- Название (Квартплата, Интернет, Netflix...)
- Сумма
- Дата списания (день месяца)
- Категория
- Активен/неактивен

**Логика:**
- Программа планирует эти расходы заранее
- За 3 дня — оповещение: "Скоро платёж..."
- При наступлении даты — подтверждение или корректировка

**Пример UI:**
```
📅 Подписки
Квартплата     | 5-го числа  | 5 000 ₽/мес  | ✅
Интернет       | 15-го числа | 600 ₽/мес    | ✅
Netflix        | 20-го числа | 499 ₽/мес    | ⚠️ Скоро
```

### Структура БД (новая таблица)

```python
# Таблица: subscriptions (подписки)

- name          # "Квартплата"

- last_paid     # 2026-03-05
- created_at
```

### Реализация (этапы)

**Этап 1 — Базовый бюджет:**
- Вкладка "Умный бюджет"

- Установка дневного/месячного лимита

**Этап 2 — Шапка:**
- Баланс с учётом бюджета
- Индикатор (🟢/🔴)

**Этап 3 — Оповещения:**
- Напоминание при превышении дневного лимита

**Этап 4 — Подписки:**
