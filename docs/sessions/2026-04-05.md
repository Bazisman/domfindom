### 7. Баг прогноза планирования

**Сценарий:** пользователь добавил фактический расход по категории с бюджетом, после чего прогноз на конец месяца уменьшился повторно.

**Причина:** в прогнозе одновременно участвовали:
- текущий баланс, уже уменьшенный фактическим расходом
- и весь бюджет месяца целиком

**Исправление:**
- бюджет в прогнозе теперь учитывается как остаток бюджета, а не как полный лимит месяца

**Вывод:** при расчётах прогноза всегда проверять, не учитывается ли одна и та же трата через разные механизмы дважды.

### 16. Исправление редактирования повторяющегося шаблона

Во время живого запуска приложения проявилась ошибка при редактировании recurring-шаблона:
- после изменения шаблона возникал `sqlite3.IntegrityError: NOT NULL constraint failed: transactions.category`

**Причина:**
- форма редактирования подставляла в поле категории не категорию, а имя шаблона
- из-за этого при сохранении мог получаться `category_id=None`
- затем генерация planned-транзакций пыталась вставить запись без корректной категории
- дополнительно плановые транзакции перегенерировались лишний раз в нескольких местах

**Исправление:**
- в `views/planning_view.py` форма редактирования теперь использует реальную `category_name`
- в `core.get_recurring_templates()` добавлен `JOIN` с `categories`, чтобы UI всегда получал имя категории
- в `core.update_recurring_template()` нормализован `template_type -> type` и регенерация выполняется централизованно
- в `services/transaction_service.py` убрана лишняя двойная генерация planned-транзакций
- в `core.generate_planned_transactions()` добавлен безопасный fallback, если категория по ID не найдена

**Проверка:**
- пользователь повторно прогнал сценарий в приложении
- результат: "вроде всё работает"
- `python -m unittest tests.test_financial_logic -v`
- результат: `OK` (6 тестов)
