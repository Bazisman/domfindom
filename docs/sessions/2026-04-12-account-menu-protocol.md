# Сессия 2026-04-12 (Account Menu Protocol)

## Задача

- Зафиксировать в протоколе продуктовые требования пользователя перед дальнейшей реализацией.
- Подготовить согласованный план по меню аккаунта, чтобы не терять контекст и не делать спорные UI-решения.

## Контекст На Старт

- В проекте уже есть авторизация, сессии, смена/сброс пароля и экран безопасности.
- Текущая зона аккаунта в шапке визуально и UX не устраивает пользователя:
  - выглядит неаккуратно;
  - на мобильном есть риск случайного нажатия `Выйти`;
  - требуется более современная, "продуктовая" модель управления аккаунтом.

## Что Проверили

- Подтверждён пользовательский запрос на новый подход:
  - нужен единый раздел "Меню аккаунта";
  - нужны личные настройки, управление аккаунтом, смена темы;
  - нужна функция полного обнуления данных;
  - нужен механизм резервного сохранения/восстановления данных;
  - резервная копия должна быть в формате "не больше одной записи на пользователя".

## Решения По Логике (Зафиксировано)

1. Меню аккаунта становится отдельным UX-центром, а не набором кнопок в строке навигации.
2. Кнопка `Выйти` убирается из зоны частого случайного тапа:
   - доступна из меню аккаунта;
   - требует подтверждение перед выполнением.
3. Добавляется блок "Личные настройки":
   - тема: `Светлая`, `Тёмная`, `Как в системе`;
   - дополнительные персональные параметры (поэтапно).
4. Добавляется блок "Управление аккаунтом":
   - профиль/безопасность/сессии;
   - действия с повышенной чувствительностью через подтверждение.
5. Добавляется блок "Мои данные":
   - `Сохранить данные` в 1 персональный слот (перезапись предыдущей копии);
   - `Восстановить данные` из этого слота после подтверждения.
6. Добавляется "Обнуление данных":
   - очищаются пользовательские финансовые данные;
   - аккаунт и доступ остаются;
   - требуется усиленное подтверждение (двойной шаг).

## Предварительная Техническая Модель

- `user_preferences`:
  - `user_id`, `theme`, `timezone`, `currency`, `updated_at`.
- `user_backup_slot` (одна запись на пользователя):
  - `user_id UNIQUE`, `payload`, `checksum`, `created_at`, `updated_at`.
- API-контур (черновой):
  - `GET/PUT /api/v1/account/preferences`
  - `POST /api/v1/account/backup/save`
  - `POST /api/v1/account/backup/restore`
  - `POST /api/v1/account/reset-all`

## Риски И Наблюдения

- Главный UX-риск: случайный запуск разрушительных операций на мобильных устройствах.
- Главный security-риск: восстановление/обнуление без дополнительной проверки контекста сессии.
- Главный data-риск: повреждение данных при восстановлении без валидации backup payload.

## Следующий Шаг

1. Согласовать финальную UX-схему меню аккаунта:
   - где расположено;
   - как открывается на desktop/mobile;
   - как оформлены чувствительные действия (`Выйти`, `Обнуление`, `Восстановление`).
2. После согласования UX перейти к поэтапной реализации:
   - этап A: UI каркас меню + тема;
   - этап B: backup-slot (1 копия);
   - этап C: безопасное обнуление + восстановление.
