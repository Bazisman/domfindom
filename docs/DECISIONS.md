# Принятые решения

> Обновлено: 2026-04-25
> Назначение: хранить продуктовые и технические договоренности, чтобы не обсуждать их заново в каждой сессии.

## Формат записи

Для каждого решения фиксируем:
- что решили;
- зачем;
- где это влияет на код или продукт.

## Решения

### 2026-04-25: миграции пользовательских SQLite-БД должны выполняться для существующих баз

Что решили:
- `auth_service.ensure_user_finance_db()` должен прогонять `core.init_db()` для существующей пользовательской `finance.db` при первом доступе в процессе;
- schema-релизы должны иметь тест, который создает старую базу без новой колонки/таблицы и проверяет автоматическую миграцию;
- после production schema-релиза нужно проверять не только `health`, но и живой write-сценарий измененного экрана.

Зачем:
- production хранит финансовые данные в отдельных `data/users/*/finance.db`;
- новые таблицы/колонки появляются в коде сразу, но старые пользовательские БД не меняются сами по себе;
- 2026-04-25 создание транзакции падало из-за отсутствующей колонки `transactions.money_source`, хотя `health` отвечал `{"status":"ok"}`.

Где влияет:
- `backend/auth/service.py`;
- `core_bootstrap.py` и доменные миграции;
- `tests/test_web_api.py`;
- `docs/DEPLOYMENT_GUIDE.md`;
- production-checklist после релиза.

### 2026-04-24: категории должны синхронизироваться по смыслу, а не по названию

Что решили:
- в будущей модели отделять смысл категории от ее личного или семейного отображаемого имени;
- использовать смысловые шаблоны (`semantic_key`) как устойчивый слой для личных и семейных категорий;
- семейные отчеты, бюджеты и forecast в целевой модели должны группировать данные через подтвержденную связь категории со смыслом, а не только через текстовое название.

Зачем:
- в семье разные участники могут называть один смысл по-разному: `Продукты`, `Еда`, `Магазин`;
- переименование не должно ломать историю и расчеты;
- аудит и слияние должны находить ошибки до активного пользования, а не после расхождения цифр.

Где влияет:
- `docs/CATEGORY_SYNC_SPEC.md`;
- будущие изменения `categories`, `transactions`, семейных отчетов, budget/forecast-логики и UI аудита категорий.

### 2026-04-24: подтверждение связи категории не равно слиянию истории

Что решили:
- preview связи категории может быть доступен участникам семьи;
- применить связь может только владелец семьи;
- применение создает связь локальной категории с семейным смыслом, но не переименовывает личную категорию и не переписывает транзакции, бюджеты или регулярные шаблоны.

Зачем:
- семье нужен общий смысл для отчетов и будущих прогнозов;
- личная история участников должна оставаться безопасной и объяснимой;
- destructive merge должен оставаться отдельным будущим сценарием с собственным preview, audit log и rollback-планом.

Где влияет:
- `backend/auth/service.py`;
- `backend/api/routes/families.py`;
- `frontend/src/pages/FamilyPage.tsx`;
- будущий перевод семейных отчетов и forecast на confirmed bindings.

### 2026-04-05: основной продукт проекта — web-приложение

Что решили:
- считать web-версию основным продуктом;
- desktop-часть считать историческим и вспомогательным слоем.

Зачем:
- пользователю нужен доступ с телефона и компьютера;
- дальнейший рост продукта логичнее вести через API и web UI.

Где влияет:
- roadmap;
- приоритет задач;
- архитектурные решения;
- документация.

### 2026-04-05: документация обязательна, а не факультативна

Что решили:
- текущее состояние хранить в `docs/PROJECT_STATE.md`;
- договоренности хранить в `docs/DECISIONS.md`;
- ход работы по умолчанию фиксировать в `docs/WORK_LOG.md`;
- при маркере `запротоколируй` обновлять документацию сразу.

Зачем:
- чтобы новые сессии не теряли контекст;
- чтобы решения не жили только в переписке.

Где влияет:
- весь рабочий процесс.

### 2026-04-11: все приватные данные изолируются по пользователю

Что решили:
- авторизация обязательна для приватных API;
- пользователь работает только со своими данными;
- изоляция строится через per-user routing данных.

Зачем:
- без этого нельзя безопасно давать доступ нескольким людям;
- это базовое требование для auth и семейного режима.

Где влияет:
- `backend/auth/service.py`;
- `backend/api/routes/*`;
- `tests/test_web_api.py`.

### 2026-04-12: семейный режим не равен слиянию личных данных

Что решили:
- не объединять личные счета и историю участников в одну общую сущность;
- семейный контекст строить как обзор поверх данных участников;
- в семейных лентах обязательно показывать автора операции.

Зачем:
- это ближе к ожидаемому сценарию семейной бухгалтерии;
- так меньше риск потерять личную структуру данных каждого участника.

Где влияет:
- `backend/api/routes/families.py`;
- `frontend/src/pages/FamilyPage.tsx`;
- `frontend/src/pages/DashboardPage.tsx`.

### 2026-04-12: роли семьи сокращены до трех

Что решили:
- оставить роли `owner`, `member`, `viewer`;
- пользовательские названия делать простыми и семейными.

Зачем:
- меньше путаницы;
- проще объяснять права и различия ролей.

Где влияет:
- семейный UI;
- API семей;
- документация по ролям.

### 2026-04-16: `display_name` имеет приоритет над email в UI

Что решили:
- везде, где один пользователь видит другого, сначала показывать `display_name`;
- email использовать только как fallback.

Зачем:
- email плохо подходит как человеческое имя;
- семейный режим становится понятнее и дружелюбнее.

Где влияет:
- account preferences API;
- `frontend/src/AppShellNext.tsx`;
- `frontend/src/pages/FamilyPage.tsx`;
- `frontend/src/pages/DashboardPage.tsx`.

### 2026-04-16: вкладка `Семья` должна быть доступна даже до создания семьи

Что решили:
- если пользователь выбрал совместный режим, он должен видеть путь к созданию семьи прямо в приложении;
- семейная вкладка не должна скрываться только потому, что семья еще не создана.

Зачем:
- иначе пользователь застревает и не понимает, как начать совместный сценарий.

Где влияет:
- `frontend/src/AppShellNext.tsx`;
- `frontend/src/pages/FamilyPage.tsx`.

### 2026-04-16: кодировка — критичный инвариант проекта

Что решили:
- все текстовые файлы хранить в `UTF-8` без BOM;
- перед сборкой и деплоем обязательно запускать `tools/check_encoding.py`;
- не исправлять mojibake случайной перекодировкой.

Зачем:
- в проекте много русских строк в UI, письмах и документации;
- одна ошибка кодировки сразу бьет по продукту и коммуникации с пользователем.

Где влияет:
- весь репозиторий;
- локальная работа через PowerShell и редактор;
- выкладка.

### 2026-04-16: production-схема — REG.RU Passenger + локальная сборка frontend

Что решили:
- backend выкладывать в `~/finance-app`;
- frontend собирать локально;
- `frontend/dist` выгружать как статику на сайт;
- backend перезапускать через Passenger.

Зачем:
- это текущая рабочая схема production на shared hosting;
- она уже проверена живыми деплоями.

Где влияет:
- `docs/DEPLOYMENT_GUIDE.md`;
- порядок релиза;
- hotfix-процесс.

### 2026-04-18: один живой протокол вместо россыпи одноразовых файлов

Что решили:
- основной журнал работы вести в одном файле `docs/WORK_LOG.md`;
- отдельные session-файлы держать только в архиве;
- накопленные уроки и устойчивые правила хранить в `docs/WORKING_MEMORY.md`.

Зачем:
- чтобы история проекта была последовательной и читаемой;
- чтобы следующие сессии реально опирались на накопленную память;
- чтобы не плодить много мелких файлов с разрозненным контекстом.

Где влияет:
- протоколирование;
- документацию;
- качество продолжения задач между сессиями.
