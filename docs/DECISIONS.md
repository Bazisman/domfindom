## Принятые Решения

### 2026-03-30: Основной счёт всегда `accounts.id = 1`

Зачем:
- упрощает расчёты и SQL
- снижает количество ветвлений в коде

Где влияет:
- `core.py`
- расчёт баланса
- переводы и корректировки

### 2026-03-30: Счета капитала живут отдельно от основного счёта

Зачем:
- капитал — отдельная сущность, а не часть обычных расходов
- проще считать автoотчисления и переводы

Где влияет:
- `capital_accounts`
- `transfers`
- `add_income_with_capital()`

### 2026-04-04: Плановые транзакции не должны попадать в фактическую статистику

Решение:
- во всех фактических расчётах нужно учитывать только
  `status = 'actual' OR status IS NULL`

Зачем:
- чтобы не дублировать суммы
- чтобы месячная статистика и балансы отражали реальность, а не прогноз

Где влияет:
- `get_balance()`
- `get_income_by_category()`
- `get_expenses_by_category()`
- `sync_accounts_with_transactions()`
- любые будущие фактические отчёты

### 2026-04-05: Бюджет в прогнозе учитывается как остаток, а не как полная сумма месяца

Решение:
- прогноз конца месяца не должен вычитать весь бюджет целиком, если часть расходов уже случилась
- в прогноз попадает только оставшаяся часть бюджета на текущий месяц

Зачем:
- фактические расходы уже уменьшили текущий баланс
- если вычитать весь бюджет ещё раз, расход учитывается дважды

Где влияет:
- `core.get_projected_balance()`
- `views/main_view.py`
- `views/planning_view.py`

### 2026-04-05: Маркер `запротоколируй` означает промежуточное сохранение контекста

Решение:

Зачем:
- удобно сохранять состояние проекта посреди длинной работы
- помогает не терять контекст при смене чата, модели или паузе

Где влияет:

### 2026-04-05: UI-лимиты и экспорт должны быть разделены

Решение:
- ограничение количества записей для UI допустимо ради производительности
- экспорт и служебные операции не должны использовать этот же лимит

Зачем:
- пользовательский интерфейс и экспорт преследуют разные цели
- иначе появляются скрытые потери данных без явной ошибки

Где влияет:

### 2026-04-05: Критическая финансовая логика должна иметь автопроверки

### 2026-04-05: Бюджетные отчёты и проверки используют только фактические расходы

Решение:
- в отчётах бюджета и проверках превышения учитывать только
  `status = 'actual' OR status IS NULL`
- `planned` не должен влиять на факт использования бюджета до момента исполнения

Зачем:
- иначе бюджетные индикаторы начинают реагировать на ещё не произошедшие траты
- это ломает смысл разделения между планом и фактом

Где влияет:
- `core.get_budget_report()`
- `core.check_budget()`
- экран планирования и бюджетные предупреждения

### 2026-04-05: Редактирование recurring-шаблона не должно терять категорию

Решение:
- форма редактирования шаблона должна работать с реальной `category_name`, а не с названием шаблона
- обновление шаблона должно регенерировать planned-транзакции в одном месте, без двойного удаления и пересоздания
- даже при повреждённой ссылке на категорию генерация не должна падать с `NOT NULL`, а должна использовать безопасный fallback и логировать предупреждение

Зачем:
- потеря категории ломала редактирование шаблона и вызывала `sqlite3.IntegrityError: NOT NULL constraint failed: transactions.category`
- recurring-логика должна быть устойчивой даже при неполных данных

Где влияет:
- `views/planning_view.py`
- `core.get_recurring_templates()`
- `core.update_recurring_template()`
- `core.generate_planned_transactions()`
- `services/transaction_service.py`
- `tests/test_financial_logic.py`

### 2026-04-05: Вкладка планирования называется `Планирование бюджетов`, а успешные действия в ней не должны открывать модальные окна

### 2026-04-05: Стратегическая цель проекта — web-приложение для ПК и телефона

Решение:
- финальной платформой проекта считать не desktop GUI на `CustomTkinter`, а адаптивное web-приложение
- текущий desktop-интерфейс использовать как переходный рабочий клиент, источник требований и полигон для проверки логики
- архитектуру дальше развивать в сторону `API-first`: финансовая логика остаётся в Python, а UI должен переехать на web-стек

Зачем:
- пользователь хочет универсальный доступ с ПК и телефона
- дальнейшая глубокая полировка `CustomTkinter` не даст нужной гибкости, адаптивности и производительности на всех устройствах
- web-версия лучше подходит для будущих сценариев: мобильный доступ, единый интерфейс, возможная синхронизация и облачное использование

Где влияет:
- на весь roadmap проекта
- `views/` становятся временным UI-слоем
- `services/` и `core.py` становятся базой для будущего API
- дальнейшие фронтенд-решения должны оцениваться с учётом будущей web-миграции

## 2026-04-10: Production backend на REG.RU запускается через Passenger

Решение:
- для REG.RU shared hosting использовать `passenger_wsgi.py`, а не ручной долгоживущий `uvicorn` в открытом shell
- FastAPI адаптировать к Passenger через `a2wsgi.ASGIMiddleware`
- production-приложение держать в `backend/site_app.py`, где `/api` обслуживается backend-роутами, а остальное может отдавать React frontend

Зачем:
- ручной запуск `python run_web_backend.py` работает только пока открыт терминал
- `nohup`/cron считаются временным костылем, а не нормальным production-режимом
- Passenger является штатным механизмом REG.RU для постоянного Python-приложения на shared-хостинге

Где влияет:
- `passenger_wsgi.py`
- `backend/site_app.py`
- `requirements-web.txt`
- `docs/DEPLOYMENT_GUIDE.md`
- production-деплой `domfindom.ru`
