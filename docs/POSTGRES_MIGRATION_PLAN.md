# План миграции на PostgreSQL

> Обновлено: 2026-04-25
> Статус: живой технический план. Не является командой к немедленной миграции production.

## 1. Зачем переходить

Текущий production работает на связке:
- `auth.db` для пользователей, сессий, семей, ролей и семейных связей категорий;
- отдельные `data/users/*/finance.db` для финансовых данных каждого пользователя.

Эта модель помогла быстро и безопасно внедрить авторизацию: у каждого пользователя свой SQLite-файл. Но для дальнейшего роста она становится дорогой:
- семейные агрегаты требуют обходить несколько пользовательских БД;
- кэш и контекст БД нужно постоянно держать под контролем;
- schema-релизы должны мигрировать много старых файлов;
- category bindings и семейные бюджеты уже создают связи между пользователями;
- будущая privacy-модель с ключами и аудитом доступа проще проектируется в единой транзакционной БД.

PostgreSQL нужен не ради моды, а как более надежная база для многопользовательского web-продукта.

## 2. Что нельзя ломать

Инварианты миграции:
- нулевая потеря пользовательских данных;
- личный и семейный режимы не смешиваются случайно;
- семейные данные остаются агрегированным обзором, а не грубым слиянием личной истории;
- `display_name` продолжает иметь приоритет над email;
- текущие расчеты forecast, budget status, accounts, capital и recurring должны совпадать до и после миграции;
- старые SQLite snapshot остаются резервной копией на период стабилизации;
- production cutover выполняется только после dry-run и сверки данных.

## 4. Целевая модель PostgreSQL

Одна PostgreSQL-БД с явной принадлежностью данных:
- все пользовательские финансовые таблицы получают `user_id`;
- семейные таблицы получают `family_id`;
- связи между семейными и личными категориями остаются явными binding-записями;
- все запросы приватных API фильтруются по текущему пользователю или его семейным правам;
- будущий слой шифрования получает отдельные таблицы/поля для metadata ключей и support grants.

Важно:
- переход на PostgreSQL не должен автоматически менять продуктовую модель семьи;
- семейный режим по-прежнему является обзором поверх данных участников;
- в PostgreSQL это будет логическая изоляция через `user_id`, роли, индексы и тесты доступа, а не физическая изоляция файлами.

## 5. Рекомендуемый стек

Минимальный целевой стек:
- PostgreSQL 15+;
- SQLAlchemy 2.x Core/ORM как слой доступа к данным;
- Alembic для миграций схемы;
- `psycopg` или `asyncpg` после выбора sync/async стратегии;
- отдельный ETL-скрипт переноса SQLite -> PostgreSQL;
- отдельный reconciliation/check скрипт после переноса.

Для текущего кода безопаснее начинать не с большой переписи `core.py`, а с переходного storage/repository слоя вокруг текущих операций.

## 6. Пошаговый план

### Этап 0. Инвентаризация схемы

Цель:
- получить точную карту текущих таблиц, колонок, индексов и фактических данных production.

Задачи:
- использовать или расширить `tools/sqlite_inventory.py` для read-only schema dump `auth.db`, исторического корневого `finance.db` при наличии и всех `data/users/*/finance.db`;
- собрать количество строк по таблицам;
- найти расхождения схем между пользовательскими БД;
- отдельно проверить наличие всех миграционных колонок (`money_source`, recurring/status поля и т.д.).

Критерий готовности:
- есть отчет `auth.db + users finance.db` без изменения данных.

### Этап 1. Целевая ER-модель

Цель:
- описать PostgreSQL-схему до написания миграций.

Задачи:
- спроектировать таблицы identity/auth;
- спроектировать таблицы finance с `user_id`;
- спроектировать семейные таблицы и category bindings;
- определить типы денег, дат, статусов и ключей;
- выбрать, какие поля будут сразу готовы к будущему шифрованию.

Критерий готовности:
- есть согласованная карта SQLite -> PostgreSQL.

### Этап 2. Миграции Alembic

Цель:
- создать повторяемую схему PostgreSQL.

Задачи:
- добавить зависимости PostgreSQL/Alembic только после утверждения схемы;
- создать начальную миграцию;
- создать индексы для частых сценариев: операции пользователя за период, planned/actual, категории и бюджеты, family bindings, auth/session lookup.

Критерий готовности:
- пустая PostgreSQL-БД поднимается из миграций локально.

### Этап 3. ETL dry-run

Цель:
- переносить данные повторяемо и проверяемо.

Задачи:
- перенести `auth.db`;
- для каждого `data/users/{user_id}/finance.db` перенести finance-таблицы с явным `user_id`;
- сохранить старые локальные id там, где они нужны для связей;
- сохранить mapping `sqlite_user_id/local_id -> postgres_id`;
- не менять исходные SQLite-файлы.

Критерий готовности:
- dry-run можно запускать повторно на копии production-данных.

### Этап 4. Сверка после переноса

Обязательные проверки:
- количество строк по каждой таблице;
- суммы доходов/расходов по пользователю и месяцу;
- balances `Безнал`/`Наличные`;
- капитал и переводы;
- recurring templates и planned-операции;
- семейные membership/roles;
- семейный dashboard/forecast/budget status для реальной семьи;
- category bindings и audit resolutions.

Критерий готовности:
- расхождения либо равны нулю, либо объяснены и зафиксированы.

### Этап 5. Переходный data-access слой

Цель:
- не переписывать весь продукт одним коммитом.

Варианты:
- сначала реализовать PostgreSQL-адаптер для нескольких read-only сценариев;
- затем mutating-сценарии;
- затем семейные агрегаты;
- только после parity переключать production.

Правило:
- один endpoint не должен в одном production-режиме писать одновременно в SQLite и PostgreSQL без отдельного dual-write протокола и сверки.

### Этап 6. Stage

Цель:
- проверить приложение как пользователь, но на PostgreSQL.

Задачи:
- поднять stage с копией production-данных;
- прогнать backend-тесты;
- собрать frontend;
- вручную проверить login, операции, planning/forecast, family dashboard, budget status, accounts/capital и category audit/bindings.

### Этап 7. Production cutover

Порядок:
1. Объявить окно обслуживания.
2. Остановить write-доступ или включить maintenance mode.
3. Сделать backup `auth.db` и всех `data/users/*/finance.db`.
4. Запустить финальный ETL.
5. Запустить сверку.
6. Переключить backend config на PostgreSQL.
7. Перезапустить Passenger.
8. Проверить health.
9. Проверить живой write-сценарий.
10. Проверить семейный сценарий.

Rollback:
- вернуть config на SQLite;
- вернуть backend к предыдущему коммиту при необходимости;
- оставить PostgreSQL-копию для разбора, не удаляя SQLite snapshot.

## 7. Связь с privacy-моделью

PostgreSQL-переход должен сразу учитывать будущие таблицы или поля:
- `user_data_keys`;
- `key_encryption_metadata`;
- `support_access_grants`;
- `support_access_audit`;
- версии ключей и ротацию.

Но сильное пользовательское шифрование нельзя внедрять в тот же момент, что и миграцию БД.

Безопасный порядок:
1. PostgreSQL-схема и перенос без изменения смысла данных.
2. Application-level encryption для чувствительных полей.
3. User-held key mode.
4. Support access grant.

## 8. Антиошибки

- Не делать big bang rewrite backend + БД + криптографию одновременно.
- Не начинать production cutover без dry-run на копии данных.
- Не считать `health` достаточной проверкой.
- Не терять старые локальные id без mapping-таблицы.
- Не ломать семейные category bindings при переносе.
- Не смешивать личные и семейные данные только потому, что PostgreSQL теперь общий.
- Не обещать пользователям “абсолютную невидимость данных” до внедрения user-held key.

## 9. Первый безопасный следующий шаг

Сделать read-only инвентаризацию:
- схема `auth.db`;
- схема исторического корневого `finance.db`, если он еще нужен для сверки или legacy-данных;
- схема всех пользовательских `finance.db`;
- количество строк;
- список расхождений схем.

Этот шаг ничего не меняет в данных и даст основу для PostgreSQL ER-модели.
