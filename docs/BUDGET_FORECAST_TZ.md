# ТЗ: Уточнение логики бюджетов и прогноза остатка

> Создано: 2026-04-22
> Статус: draft
> Цель: сделать прогноз остатка на конец месяца реалистичным для дневных бюджетов и не смешивать плановый лимит с прогнозом фактических трат.

## 1. Проблема

Сейчас в приложении бюджет участвует в прогнозе слишком консервативно:

- для дневных бюджетов в расчет попадает полный месячный эквивалент лимита;
- из-за этого прогноз остатка на конец месяца может быть занижен;
- если пользователь тратит меньше дневного лимита, сэкономленная часть уже прошедших дней теряется в прогнозе;
- в интерфейсе показывается один `remaining`, из-за чего смешиваются два разных смысла:
  - остаток по лимиту;
  - прогноз того, сколько денег реально останется к концу месяца.

Пример:

- бюджет "Продукты": `2350` в день;
- условный месячный лимит: `70500`;
- пользователь тратил меньше плана;
- фактический ожидаемый расход к концу месяца может быть, например, `65300`;
- если приложение продолжает вычитать весь лимит `70500`, прогноз показывает меньше денег, чем пользователь реально увидит в конце месяца.

## 2. Цель изменения

Нужно разделить два разных показателя:

- `Остаток по плану` — сколько лимита еще доступно относительно полного бюджета периода;
- `Прогнозируемый остаток` — сколько денег, вероятно, останется к концу месяца с учетом уже случившейся экономии.

Принцип:

- для `daily` бюджета прогноз должен учитывать, что прошедшие дни уже закрыты фактическими расходами, а не полным планом;
- для `monthly` бюджета прогноз пока не усложняем и не пытаемся угадывать темп;
- недельный период в этом ТЗ не вводим.

## 3. Решение верхнего уровня

### 3.1. Для дневного бюджета

Храним текущую модель бюджета как есть:

- пользователь задает `amount` и `period = daily`;
- месячный эквивалент бюджета остается нужен для статуса бюджета, проверки превышения и отчетов.

Добавляем отдельную прогнозную логику:

- `месячный лимит = daily_amount * days_in_month`
- `плановый расход на текущую дату = daily_amount * day_of_month`
- `факт на текущую дату = actual_spent_to_date`
- `экономия к текущей дате = max(0, plan_to_date - fact_to_date)`
- `будущий лимит = daily_amount * remaining_days_in_month`
- `прогноз расхода до конца месяца = fact_to_date + future_limit`
- `прогнозируемый остаток бюджета = monthly_limit - forecast_spend_to_month_end`

Эквивалентная форма:

- `прогнозируемый остаток бюджета = monthly_limit - fact_to_date - future_limit`
- это то же самое, что уже накопленная экономия к текущей дате.

Смысл:

- прошлые дни считаются по факту;
- будущие дни считаются по лимиту;
- экономия в уже прошедших днях не теряется.

### 3.2. Для месячного бюджета

Пока оставляем базовую модель без прогнозирования по темпу.

Используем только:

- `остаток по плану = monthly_limit - spent`

Не вводим отдельный "умный прогноз" для месячных бюджетов на этом этапе, потому что:

- у месячного лимита нет встроенного ритма потребления;
- попытка экстраполировать по факту будет часто ошибаться для нерегулярных расходов;
- пользователь сам подтвердил, что на данном этапе месячные бюджеты можно оставить как есть.

### 3.3. Что не делаем в этом ТЗ

- не добавляем недельный период;
- не добавляем настройку "сколько раз в месяц бывает расход";
- не меняем модель хранения бюджета в БД;
- не делаем ML, автообучение или прогноз по истории.

Идея с "событиями в месяц" зафиксирована как возможное следующее улучшение, но не входит в текущий объем.

## 4. Требования к расчетам

### 4.1. Остаток по плану

Для всех текущих периодов:

- `plan_remaining = max(0, budget_amount - spent)`

Где:

- `budget_amount` — текущий лимит периода, уже приведенный к месячному эквиваленту для `daily`;
- `spent` — фактические расходы за месяц;
- `planned` транзакции не участвуют.

### 4.2. Прогнозируемый остаток для daily

Только для бюджетов с `period = daily`:

- `days_in_month` — число дней в текущем месяце;
- `current_day` — номер текущего дня месяца;
- `remaining_days = max(0, days_in_month - current_day)`;
- `monthly_limit = daily_amount * days_in_month`;
- `future_limit = daily_amount * remaining_days`;
- `forecast_spend = spent + future_limit`;
- `forecast_remaining = max(0, monthly_limit - forecast_spend)`.

Важно:

- `spent` — только фактические расходы с начала месяца по текущую дату;
- если расход уже превысил допустимый уровень, `forecast_remaining` может быть `0`;
- отрицательные значения наружу не выводим, если UI не требует отдельного показа перерасхода.

### 4.3. Прогнозируемый остаток для monthly

На первом этапе:

- `forecast_remaining = null`
- либо
- `forecast_remaining = plan_remaining`, но только если в UI это явно помечено как отсутствие отдельного прогноза.

Предпочтительный вариант:

- возвращать отдельный признак, что прогноз не рассчитывается;
- в UI показывать только `Остаток по плану` без второй строки.

## 5. Изменения в backend

### 5.1. Новые поля в статусе бюджета

Нужно расширить `BudgetStatusItem` новыми полями.

Текущие поля:

- `budget_amount`
- `spent`
- `remaining`
- `percent`
- `over_budget`

Предлагаемые новые поля:

- `plan_remaining: float`
- `forecast_remaining: float | null`
- `forecast_mode: str`

Возможные значения `forecast_mode`:

- `none` — прогноз отдельно не считается;
- `daily_tempo` — прогноз рассчитан по логике "факт + лимит на оставшиеся дни".

Правило совместимости:

- `remaining` на переходном этапе можно оставить как alias к `plan_remaining`;
- новый frontend должен постепенно перейти на явное использование `plan_remaining`.

### 5.2. Обновление схем

Понадобится обновить:

- `backend/schemas/budgets.py`
- при необходимости `backend/schemas/dashboard.py`
- `frontend/src/lib/api.ts`

### 5.3. Логика статуса бюджета

Понадобится обновить расчет в цепочке:

- `core.get_budget_status()`
- нижележащую реализацию `_budgets_get_budget_status(...)`

Требование:

- текущее поведение `budget_amount`, `spent`, `percent`, `over_budget` не ломаем;
- добавляем поверх него вычисление `plan_remaining`, `forecast_remaining`, `forecast_mode`.

### 5.4. Логика прогноза на конец месяца

Понадобится обновить расчет в цепочке:

- `core.get_projected_balance()`
- нижележащую реализацию `_forecast_get_projected_balance(...)`

Новое правило:

- в общий `projected_balance` нельзя механически подставлять весь оставшийся месячный лимит для `daily`, если часть периода уже пройдена экономно;
- для `daily` в `budget_remaining` должен участвовать прогнозный остаток по новой формуле;
- для `monthly` поведение остается текущим.

Уточнение по именам:

- если текущее поле `budget_remaining` в forecast на самом деле станет означать "суммарный прогнозируемый остаток бюджета", это нужно явно описать в API и UI;
- если поле уже используется как "остаток по плану", лучше ввести новое поле, чтобы не ломать смысл.

Предпочтительный вариант:

- оставить существующее поле совместимым;
- добавить новое поле прогноза, например `budget_forecast_remaining`.

## 6. Изменения в frontend

### 6.1. Экран планирования

На карточке бюджета вместо одного безымянного остатка нужно показывать явную семантику.

Для `daily`:

- `Потрачено X из Y`
- `Остаток по плану: ...`
- `Прогнозируемый остаток: ...`

Для `monthly`:

- `Потрачено X из Y`
- `Остаток по плану: ...`

Если отдельный прогноз не рассчитывается:

- строка `Прогнозируемый остаток` не показывается;
- либо показывается подпись `Прогноз не рассчитывается для месячного лимита`.

### 6.2. Экран общего прогноза

Если на экране "Прогноз на конец месяца" используется агрегированный бюджетный остаток, пользователь должен видеть понятную формулировку:

- `Остаток с учетом бюджетов`
- или
- `Прогноз с учетом бюджетных лимитов`

Нельзя оставлять логику, в которой UI визуально обещает "реальный прогноз", но backend фактически вычитает весь лимит для уже прошедших дней.

## 7. UX-требования

- пользователь должен различать "что еще можно потратить по плану" и "сколько, вероятно, останется";
- подписи должны быть простыми и русскоязычными;
- на карточке не должно быть двусмысленного единственного слова `Осталось`, если рядом нет уточнения;
- для месячного бюджета интерфейс не должен симулировать точность, которой у нас нет.

Рекомендуемые подписи:

- `Остаток по плану`
- `Прогнозируемый остаток`
- `Прогноз по дневному лимиту`

## 8. Примеры

### 8.1. Daily бюджет, пользователь экономит

Дано:

- `2350` в день;
- в месяце `30` дней;
- сегодня `10` число;
- факт расходов `18000`.

Расчет:

- `monthly_limit = 2350 * 30 = 70500`
- `remaining_days = 20`
- `future_limit = 2350 * 20 = 47000`
- `forecast_spend = 18000 + 47000 = 65000`
- `plan_remaining = 70500 - 18000 = 52500`
- `forecast_remaining = 70500 - 65000 = 5500`

Показ:

- `Остаток по плану: 52 500`
- `Прогнозируемый остаток: 5 500`

### 8.2. Monthly бюджет

Дано:

- лимит `70000`;
- факт `18000`.

Расчет:

- `plan_remaining = 70000 - 18000 = 52000`
- отдельный прогноз не строим.

Показ:

- `Остаток по плану: 52 000`

## 10. Риски

- путаница из-за сохранения старого поля `remaining`;
- риск сломать текущие тесты, завязанные на старый смысл `budget_remaining`;
- риск UX-перегруза, если вывести слишком много цифр сразу;
- риск неверного агрегирования в семейном режиме, если часть данных считается по старой логике, а часть по новой.

## 12. Критерии приемки

- для `daily` бюджета приложение перестает занижать прогноз остатка на конец месяца из-за полного вычитания уже "сэкономленных" дней;
- пользователь видит отдельно `Остаток по плану` и `Прогнозируемый остаток`;
- для `monthly` бюджета не появляется ложный "умный прогноз";
- существующие базовые сценарии бюджетов и семейного режима остаются рабочими;
- новые расчеты покрыты тестами.
