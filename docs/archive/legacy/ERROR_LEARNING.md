## 2026-03-30: invalid color name "#gray90"

**Проблема:** Ошибка `invalid color name "#gray90"` при запуске приложения

**Причина:** CustomTkinter использует имена цветов (`gray90`), а не hex (`#gray90`). Лишние `#` ломают парсинг.

**Решение:**
```python
# ✗ Неправильно
fg_color="#gray90"

# ✓ Правильно
fg_color="gray90"
# или для темной/светлой темы
fg_color=("gray10", "gray90")
```

**Как избежать:** Запомнить — CTk использует имена цветов, не hex. Всегда проверять цвета в документации CTk.

## 2026-03-30: Окно сверки не помещалось на экране

**Проблема:** Кнопки "Пересчитать", "История" были за пределами окна (geometry="450x500")

**Причина:** Не хватало высоты для всех элементов

**Решение:**
```python
# Увеличил высоту и сделал растягиваемым
self.geometry("450x600")
self.resizable(True, True)
```

**Как избежать:** При добавлении новых элементов в окно — проверять что всё помещается, тестировать на разных размерах.

## 2026-03-30: Сумма в источниках сверки не сохранялась

**Проблема:** При изменении суммы и переключении фокуса значение сбрасывалось

**Причина:** Не был привязан обработчик `<FocusOut>` для сохранения в БД

**Решение:**
```python
# Добавить привязку события
entry.bind("<FocusOut>", lambda e, sid=source_id, ent=entry: self.save_source_balance(sid, ent.get()))
entry.bind("<KeyRelease>", lambda e: self.update_total())
```

**Как избежать:** При создании редактируемых полей — сразу добавлять сохранение при потере фокуса.

## 2026-03-29: Вкладка показывала "В разработке"

**Проблема:** При открытии вкладки появлялось сообщение "В разработке" вместо контента

**Причина:** Исключение в `__init__` вкладки, но оно не показывалось пользователю

**Решение:**
- Добавил try-except с traceback для отладки
- Проверяю синтаксис перед запуском: `python -c "from views.XXX import YYY"`

**Как избежать:** Всегда проверять синтаксис после изменений. Использовать traceback при отладке.

## 2026-03-29: Пропущен return в методе

**Проблема:** Функция продолжала выполняться после возврата результата

**Причина:** Не было `return` после вызова функции

**Решение:**
```python
# ✗ Неправильно
def open_reconcile_window(self):
    self.reconcile_window = ReconcileWindowCompact(self)

# ✓ Правильно  
def open_reconcile_window(self):
    self.reconcile_window = ReconcileWindowCompact(self)
    return  # если далее есть код
```

**Как избежать:** Проверять структуру метода, особенно если после вызова идёт другой код.

## 2026-03-28: Дублирующиеся записи в БД

**Проблема:** При экспорте/импорте появлялись дубликаты переводов

**Причина:**
- Экспорт сохранял удалённые переводы (is_active=0)
- Импорт создавал записи с is_active=NULL

**Решение:**
```python
# Экспорт — только активные
cursor.execute("SELECT * FROM transfers WHERE is_active = 1")

# Импорт — явно указывать is_active
cursor.execute("INSERT INTO transfers (...) VALUES (..., 1)")
```

**Как избежать:** Всегда фильтровать по is_active=1 при чтении данных. Всегда указывать is_active при вставке.

## 2026-04-05: Двойной учёт бюджета в прогнозе планирования

**Проблема:** После фактического расхода прогноз на конец месяца уменьшался повторно, хотя бюджет уже учитывал эту трату в планировании.

**Причина:** Формула прогноза вычитала:
- текущий баланс, который уже уменьшился из-за фактического расхода
- и весь бюджет месяца целиком

Из-за этого одна и та же трата влияла на прогноз дважды.

**Решение:**
```python
# ✗ Неправильно
projected = current_balance + planned_income - planned_expense - total_budget

# ✓ Правильно
# в прогноз вычитать только оставшуюся часть бюджета
budget_remaining = max(total_budgets - current_expenses, 0)
projected = current_balance + planned_income - planned_expense - budget_remaining
```

**Как избежать:** Если расчёт уже использует `current_balance`, всегда проверять, не зашиты ли те же фактические расходы второй раз через бюджеты, прогнозы или агрегаты.

## Частые причины ошибок

1. **Не проверять синтаксис** → Всегда `python -c "from X import Y"`
2. **Не читать существующий код** → Сначала найди похожее
3. **Не тестировать** → Запусти приложение
4. **Не проверять логи** → Смотри output.log
5. **Не учиться на ошибках** → Записывай в этот файл
