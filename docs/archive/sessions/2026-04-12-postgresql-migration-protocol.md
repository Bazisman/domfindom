# Сессия 2026-04-12 (PostgreSQL Migration Protocol)

## Решение

Перед масштабной реализацией семейного режима целесообразно перейти с SQLite на PostgreSQL.

Причина:

- семейный режим предполагает несколько активных пользователей;
- одновременные операции и ролевой доступ требуют более надежной конкурентной модели;
- PostgreSQL лучше подходит для транзакционности, блокировок, индексов и масштабирования.

## Оценка сложности

- Сложность: средняя.
- Ключевой риск: корректная миграция пользовательских данных без потерь и рассинхронизации.

## Почему не откладывать

- Внедрение семейного функционала сильно увеличит связанность данных.
- Миграция после запуска сложных семейных сценариев будет дороже и рискованнее.

## Рекомендуемый порядок работ

1. Подготовить слой доступа к данным (репозитории/ORM), чтобы минимизировать привязку к SQLite.
2. Развернуть PostgreSQL окружение (dev/stage/prod).
3. Описать схему и миграции (Alembic).
4. Реализовать ETL-скрипт переноса данных из SQLite в PostgreSQL.
5. Добавить сверку данных после миграции:
- количество записей;
- контрольные суммы по ключевым таблицам;
- сверка финансовых агрегатов.
6. Прогнать тестирование в stage.
7. Выполнить переключение production на PostgreSQL в окно обслуживания.
8. Оставить SQLite snapshot как резервный backup на период стабилизации.

## Минимальные требования к качеству переключения

- Нулевая потеря данных.
- Повторяемый скрипт миграции.
- Возможность отката (backup + план rollback).
- Мониторинг ошибок после переключения.

## Следующий практический шаг

Подготовить технический план реализации перехода:

- выбор ORM/подхода;
- карта таблиц SQLite -> PostgreSQL;
- структура миграций Alembic;
- сценарий dry-run и боевого cutover.
