# Протокол: стабильная кодировка UTF-8 (PowerShell + проект)

Дата: 2026-04-12  
Статус: принят, обязателен к соблюдению

## Что было причиной проблемы
- Файлы могли сохраняться с «моджибейком» (`РЎРµ...`) при правках через shell-перезапись.
- В Windows PowerShell 5.1 запись через `Set-Content -Encoding utf8` добавляет BOM.
- BOM и/или уже испорченные строки попадали в сборку и отображались на сайте.

## Проверенные факты
- Профиль PowerShell настроен на UTF-8:
  - `InputEncoding = utf-8`
  - `OutputEncoding = utf-8`
  - `chcp 65001`
- Для записи без BOM работает функция:
  - `Set-Utf8NoBomFile`
- Проверка проекта:
  - `python tools/check_encoding.py --root .` — обязательная перед сборкой/деплоем.

## Обязательные правила работы
1. Правки `.ts/.tsx/.py/.css/.md/.json` делать через IDE или `apply_patch`.
2. Не использовать массовые shell-перезаписи с кириллицей (`Set-Content`, `Out-File`) для исходников.
3. Если запись через PowerShell неизбежна — только через `Set-Utf8NoBomFile`.
4. Перед `npm run build`/деплоем всегда запускать:
   - `python tools/check_encoding.py --root .`
5. Если найдены строки вида `РЎР...` — считать это инцидентом кодировки и исправлять до деплоя.

## Операционный чеклист перед выкладкой
1. `python tools/check_encoding.py --root .`
2. `npm run build`
3. Проверить, что `dist/index.html` ссылается на актуальный `assets/index-*.js`.
4. После загрузки на хост — проверить `index.html` на сервере и активный asset-файл.

## Личное обязательство агента
- Всегда использовать безопасный путь правок (IDE/`apply_patch`).
- Не применять опасные для кириллицы методы записи.
- Блокировать деплой при любых признаках повреждения кодировки.
