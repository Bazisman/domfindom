# Аудит проекта: фаза 1 (техдолг и рефакторинг)

Дата: 2026-04-12

## Что проверено
- Кодировка: `python tools/check_encoding.py --root .` — успешно.
- Frontend build: `npm run build` — успешно.
- Backend тесты: не выполнены, `pytest` отсутствует в текущем окружении.
- Быстрый структурный аудит по размеру/связанности модулей.

## Ключевые находки

1. Крупные "монолитные" модули (высокий риск регрессий при изменениях)
- `backend/auth/service.py` (~1235 строк)
- `frontend/src/lib/api.ts` (~667 строк)
- `frontend/src/AppShellNext.tsx` (~606 строк)
- `frontend/src/pages/PlanningPage.tsx` (~29600 байт)

2. Недостаточная наблюдаемость качества в локальном окружении
- Нет доступного `pytest` в текущем runtime, из-за этого нет быстрого регулярного backend smoke-check.
- Нет явного frontend lint-шага в scripts (только сборка + ts).

3. Проблема прав на pycache в текущем окружении
- `compileall` падал на `PermissionError` при записи `.pyc`.
- Это не критично для прода, но мешает локальному quality-loop.

## Риски при дальнейшем росте функционала
- Дальнейшее развитие семейного бюджета без декомпозиции `auth/service.py` усложнит поддержку ролей, приглашений и безопасности.
- Увеличение UI-логики в `AppShellNext.tsx` повышает риск побочных эффектов в навигации/меню/режимах.
- Отсутствие стабильного тестового контура увеличивает вероятность "тихих" регрессий.

## Рекомендованный план (безопасный, поэтапный)

### Этап A (быстрый, низкий риск)
1. Добавить рабочий backend test-runner и smoke-набор в CI/локально.
2. Добавить frontend lint/type gate (минимум eslint + текущий ts build).
3. Оформить отдельный refactor backlog с малыми PR-порциями (не "большой взрыв").

### Этап B (архитектурный рефакторинг)
1. Декомпозировать `backend/auth/service.py`:
   - preferences/session activity
   - backup/reset
   - family membership/invites
2. Декомпозировать `frontend/src/lib/api.ts` по доменам:
   - `api/auth.ts`, `api/families.ts`, `api/transactions.ts`, ...
3. Вынести из `AppShellNext.tsx` hooks:
   - `useAccountMenu`
   - `useFamilyWorkspace`
   - `useNotifications`

### Этап C (производительность и UX)
1. Стандартизовать query keys + invalidation policy.
2. Добавить ограничение/пагинацию крупных лент (семья/транзакции) по умолчанию.
3. Укрепить сценарии восстановления сессии после F5 (навигационные guard-ы).

## Решение по фазе 1
- Начать с Этапа A, затем B.
- Любой рефакторинг — только с прохождением encoding check + build + smoke.
