# WEB MIGRATION PLAN

> Обновлено: 2026-04-05
> Назначение: зафиксировать целевой web-стек и безопасный профессиональный план перехода проекта с desktop GUI на универсальное web-приложение.

---

## Цель

Перевести проект `Домашняя бухгалтерия` из desktop-приложения на `CustomTkinter` в адаптивное web-приложение, удобное:
- на ПК
- на телефоне
- в браузере без привязки к конкретной ОС

При этом сохранить и не потерять:
- финансовую логику
- накопленную документацию
- рабочие данные SQLite
- тесты критических сценариев
- продуктовые договорённости по бюджетам, капиталу, плановым операциям и прогнозам

---

## Рекомендуемый Стек

### Backend

- `FastAPI`
- `Pydantic v2`
- `SQLAlchemy 2.x` или аккуратный переходный слой поверх текущего `core.py`
- `Alembic` для миграций схемы
- `pytest` для backend-тестов

Почему:
- современный Python API-стек
- удобно строить REST API и постепенно выносить логику из desktop UI
- легко документировать контракты и валидировать данные

### Frontend

- `React 18`
- `TypeScript`
- `Vite`
- `TanStack Query`
- `React Router`
- `react-hook-form`
- `zod`

Почему:
- зрелый стек для сложных интерфейсов с формами, фильтрами и списками
- удобная адаптация под desktop и mobile
- хорошая база для постепенного роста приложения

### UI Слой

- `Tailwind CSS`
- `shadcn/ui` на базе `Radix UI`
- `Recharts` для графиков

Почему:
- позволяет быстро собрать аккуратный и современный интерфейс
- даёт хороший контроль над визуалом без тяжёлой дизайн-системы с нуля
- удобно поддерживать единый стиль на телефоне и ПК

### Mobile-Friendly Возможности

- responsive web UI как базовый режим
- `PWA` как следующий шаг
- при необходимости позже можно рассмотреть `Capacitor` для упаковки в мобильную оболочку

### Хранение Данных

Этап 1:
- оставить `SQLite`, если приложение пока single-user или локальное

Этап 2:
- при переходе к многопользовательскому или облачному сценарию планировать `PostgreSQL`

## Целевая Архитектура

### Что остаётся основой проекта

- предметная логика финансов
- правила расчёта бюджета и прогноза
- логика капитала, переводов и recurring-шаблонов
- документация решений
- сценарные тесты

### Что будет переписано

- весь UI-слой `views/`
- навигация
- формы
- экраны списков и карточек

### К чему надо прийти

- `backend/` или эквивалентный API-слой на Python
- `frontend/` как отдельное web-приложение
- UI общается только с API, а не с БД напрямую
- все финансовые формулы живут в backend-слое

---

## Переиспользование Текущего Проекта

### С высокой вероятностью сохраняем

- `core.py` как источник финансовой логики и SQL-операций на первом этапе
- `services/` как базу для сервисного слоя
- `finance.db` и текущую модель данных
- `tests/test_financial_logic.py`
- `docs/DECISIONS.md`
- `docs/PROJECT_STATE.md`
- `docs/ERROR_LEARNING.md`

### Что желательно постепенно переработать

- монолитный `core.py` разделить на более понятные доменные модули
- убрать прямую зависимость UI от внутренних методов
- формализовать DTO и API-контракты

### Что не стоит тащить в web один в один

- `CustomTkinter`-паттерны экранов
- логику refresh/redraw из desktop UI
- desktop-специфичные компромиссы производительности

## Профессиональный План Миграции

### Этап 0. Архитектурная Фиксация

Цель:
- официально принять web как целевую платформу

Результат:
- стек зафиксирован
- план миграции создан
- desktop UI признан временным клиентом

Критерий завершения:
- команда принимает `FastAPI + React + TypeScript` как базовый путь

### Этап 1. Подготовка Backend К API-First

Цель:
- отделить финансовую логику от desktop UI настолько, чтобы её можно было безопасно отдавать через HTTP API

Задачи:
- выделить API-friendly сервисный слой
- определить DTO для транзакций, категорий, бюджетов, счетов, recurring-шаблонов
- определить, какие части `core.py` можно использовать напрямую, а какие надо обернуть
- договориться о форматах дат, денег, статусов и ошибок

Результат:
- backend можно вызывать не только из desktop UI, но и через API

Риски:
- размытая граница между `services/` и `core.py`
- риск случайно утащить UI-логику в API

Критерий завершения:
- есть чистая схема backend-контрактов и список будущих эндпоинтов

### Этап 2. Базовый Backend Каркас

Цель:
- поднять первый рабочий API-контур

Задачи:
- создать `FastAPI` приложение
- подключить конфигурацию, логирование и healthcheck
- реализовать базовые эндпоинты:
  - категории
  - транзакции
  - бюджеты
  - прогноз
  - recurring-шаблоны
- добавить backend-тесты на API-слой

Результат:
- frontend уже может получать реальные данные из API

Критерий завершения:
- API покрывает минимальный пользовательский сценарий чтения и записи данных

### Этап 3. Frontend Foundation

Цель:
- создать web-клиент с базовой дизайн-системой и навигацией

Задачи:
- поднять `React + TypeScript + Vite`
- настроить роутинг
- выбрать базовую визуальную систему
- собрать layout для:
  - desktop
  - tablet
  - mobile
- настроить работу с API через `TanStack Query`

Результат:
- есть web-оболочка, в которую можно переносить реальные экраны

Критерий завершения:
- frontend собирается и уже умеет отображать данные из API

### Этап 4. Миграция По Срезам Функциональности

Цель:
- переносить систему по частям, а не переписывать всё сразу

Рекомендуемый порядок:
1. Дашборд и баланс
2. Транзакции
3. Категории
4. Планирование бюджетов
5. Капитал и переводы
6. Отчёты
7. Настройки

Для каждого среза:
- сначала API-контракты
- потом web UI
- потом ручная сверка с desktop-логикой
- потом тесты регрессии

Результат:
- приложение постепенно становится usable в web без большого разового риска

Критерий завершения:
- ключевые пользовательские сценарии работают в web-пути

### Этап 5. Миграция Эксплуатации

Цель:
- подготовить проект к реальному использованию как web-приложения

Задачи:
- решить вопрос авторизации
- определить стратегию хранения данных
- продумать backup и восстановление
- продумать deployment
- при необходимости включить PWA

Результат:
- web-версия готова к повседневному использованию

Критерий завершения:
- основная работа пользователя может проходить в web без возврата к desktop-клиенту

### Этап 6. Завершение Перехода

Цель:
- перевести desktop GUI в архивный или вспомогательный режим

Варианты:
- оставить desktop-клиент как временный fallback
- заморозить его и развивать только backend + web
- полностью снять с активной разработки

Критерий завершения:
- основная продуктовая разработка больше не зависит от `CustomTkinter`

## Риски И Как Их Снизить

### Риск: начать переписывать всё сразу

Как снижать:
- переносить по функциональным срезам
- не делать big bang rewrite

### Риск: потерять финансовую логику при переносе

Как снижать:
- держать расчёты в backend
- расширять автоматические тесты до и во время миграции

### Риск: слишком долго полировать старый desktop UI

Как снижать:
- улучшать desktop только там, где это помогает текущей эксплуатации до web-версии
- не строить в `CustomTkinter` то, что всё равно будет переписано

### Риск: рано усложнить инфраструктуру

Как снижать:
- начать с API-first монолита
- не уходить в микросервисы
- не тащить сложный cloud stack до появления реальной необходимости
