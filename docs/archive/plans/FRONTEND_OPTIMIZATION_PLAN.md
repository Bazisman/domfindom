# FRONTEND OPTIMIZATION PLAN

> Обновлено: 2026-04-05
> Назначение: безопасный пошаговый план оптимизации UI с фиксацией статуса по каждому этапу.

---

## Цели

- уменьшить подвисания при растягивании окна
- ускорить тяжёлые экраны, особенно `Категории` и `Планирование бюджетов`
- сократить количество полных перерисовок UI
- отделить обновление данных от полного перестроения виджетов

### 2. Оптимизация экрана категорий

Статус: `IN PROGRESS`

Что делаем:
- убрать избыточный full redraw в `views/categories_view.py`
- сократить число `destroy()` + повторного создания строк
- улучшить модель подгрузки категорий

Критерий проверки:
- вкладка `Категории` открывается заметно плавнее
- фильтрация и прокрутка не пересобирают весь экран без необходимости

Прогресс:
- [x] Разделить загрузку данных и отрисовку
- [x] Уменьшить полную пересборку списка
- [ ] Проверить поведение фильтров и прокрутки

Промежуточный результат:
- в `views/categories_view.py` добавлено разделение между `refresh_data()` и локальной перестройкой списка
- фильтр больше не перечитывает категории из сервиса при каждом переключении
- счётчик и список теперь обновляются из уже загруженного набора `all_categories`
- пагинация и подгрузка продолжают работать поверх локально отфильтрованного списка
- сохранение редактирования категории теперь собирает изменения в один `update_category(...)`, чтобы не вызывать каскад из нескольких refresh подряд
- создание категории теперь сразу сохраняет `color` и `icon`, без дополнительного `update_category(...)` и без второго refresh после добавления
- добавлена сигнатура первого экрана списка, чтобы повторный refresh с тем же набором категорий не вызывал бессмысленную полную перерисовку первой страницы
- уведомления `CategoryService` теперь проходят через мягкий debounce в `views/categories_view.py`, чтобы несколько быстрых сигналов склеивались в одно обновление экрана
- для `views/categories_view.py` добавлено переиспользование row-виджетов: первая страница списка теперь не уничтожается и не создаётся заново целиком при каждом полном refresh

### 3. Оптимизация экрана планирования бюджетов

Статус: `IN PROGRESS`

Что делаем:
- в `views/planning_view.py` разделяем обновление шаблонов, прогноза и бюджетов
- после локальных действий обновляем только затронутый блок

Критерий проверки:
- операции с бюджетами не должны полностью перерисовывать шаблоны без причины
- операции с шаблонами не должны полностью перерисовывать бюджеты без причины

Прогресс:
- [x] Разделить `load_data()` на более узкие обновления
- [ ] Убрать лишние полные перерисовки
- [ ] Проверить пользовательские сценарии

Промежуточный результат:
- в `views/planning_view.py` выделен отдельный `load_templates()` для блока регулярных операций
- действия с шаблонами теперь чаще обновляют только шаблоны и прогноз
- действия с бюджетами обновляют бюджеты и прогноз без лишней пересборки шаблонов
- listener-обновления проходят через debounce, а локальные точечные refresh умеют отменять лишний отложенный полный redraw
- для шаблонов, бюджетов и списка категорий в форме бюджета добавлены сигнатуры, чтобы одинаковые данные не вызывали бессмысленную полную пересборку блоков
- блок бюджетов переведён на переиспользование карточек: при обновлении карточки теперь чаще перенастраиваются, а не создаются заново
- блок регулярных операций тоже переведён на переиспользование карточек: шаблоны теперь чаще перенастраиваются, а не пересоздаются полностью
