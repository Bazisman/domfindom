# WEB MVP SCOPE

> Обновлено: 2026-04-08
> Назначение: зафиксировать минимальный полезный объём первой web-версии, чтобы миграция шла по реальному продукту, а не по всему desktop-приложению сразу.

---

## Цель Первого Web MVP

Собрать первую web-версию, которая уже позволяет:
- увидеть текущее финансовое состояние
- посмотреть последние транзакции
- добавить доход или расход
- увидеть категории
- увидеть бюджеты и прогноз конца месяца

Это не “полный перенос проекта”, а первый работающий web-контур.

---

## Что Входит В MVP

### 1. Dashboard

Обязательно:
- текущий баланс
- прогноз конца месяца
- последние транзакции
- краткий блок по бюджетам

### 2. Транзакции

Обязательно:
- список транзакций
- фильтр по периоду
- добавление транзакции
- удаление транзакции

Можно отложить:
- сложную фильтрацию
- массовые операции
- экспорт

### 3. Категории

Обязательно:
- список категорий
- отображение типа, цвета и иконки

Можно отложить:
- полный CRUD через web UI
- сложную административную работу с категориями

### 4. Бюджеты

Обязательно:
- отчёт по бюджетам
- статус бюджета по категориям

Можно отложить:
- полный редактор бюджетов в первом релизе, если backend уже умеет отдавать данные

### 5. Прогноз

Обязательно:
- блок `Баланс на конец месяца`
- базовая расшифровка прогноза

---

## Что Сознательно Не Входит В Первый MVP

- сложный экран капитала
- переводы между счетами через web UI
- recurring-шаблоны с полным редактором
- отчёты и аналитика в полном объёме
- настройки
- авторизация
- PWA/offline

---

## Минимальный Backend Для MVP

Нужны ресурсы:
- `GET /api/v1/health`
- `GET /api/v1/dashboard`
- `GET /api/v1/transactions`
- `POST /api/v1/transactions`
- `DELETE /api/v1/transactions/{id}`
- `GET /api/v1/categories`
- `GET /api/v1/budgets/report`
- `GET /api/v1/budgets/status`
- `GET /api/v1/forecast/month-end`

---

## Минимальный Frontend Для MVP

Нужны экраны:
- `/` — dashboard
- `/transactions`
- `/categories`
- `/budgets`

Для первой итерации допустимо:
- без сложного роутинга
- без авторизации
- без полноценной дизайн-системы

Но уже должно быть:
- responsive layout
- нормальная работа на телефоне
- единый стиль

---

## Критерий Готовности MVP

Web MVP можно считать состоявшимся, когда пользователь может:
- открыть приложение в браузере
- увидеть свой баланс и прогноз
- посмотреть последние операции
- добавить новый доход или расход
- увидеть категории и бюджетный статус

---

## Следующий Шаг После MVP

После этого можно переносить:
- капитал и переводы
- recurring-шаблоны
- настройки
- расширенную аналитику
