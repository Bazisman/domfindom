# Рабочая память проекта

> Обновлено: 2026-04-22
> Назначение: хранить устойчивые выводы и антиошибки, чтобы не повторять одни и те же промахи в новых сессиях.

## Как использовать

- Читать перед большими правками, рефакторингом, выкладкой и отладкой.
- Добавлять короткие устойчивые уроки, а не чат-историю.
- Предпочитать timeless-правила: что ломается, чего ждет пользователь, что нельзя повторять.
- Если новая сессия начинается командой `начинаем`, этот файл должен восприниматься как обязательная память о том, как именно нужно работать в репозитории, а не как факультативная справка.

## Продуктовая память

- Главный продукт — web-приложение, а не исторический desktop UI.
- `display_name` имеет приоритет над email в интерфейсе.
- Личный и семейный режимы — разные контексты, их нельзя смешивать случайно.
- Семейные данные — это агрегированный обзор поверх данных участников, а не наивное физическое слияние в один общий набор.
- Пользователь предпочитает практичный, понятный UX без лишней технической терминологии.
- Пользователь хочет русский текст в UI, auth-flow, письмах и системной обратной связи.

## Память по документации

- `docs/WORK_LOG.md` — основной рабочий протокол.
- Документов должно быть немного, но они должны быть живыми и актуальными.
- Старые планы, session-файлы и одноразовые материалы нужно уводить в `docs/archive/`.
- Если пользователь пишет `запротоколируй`, нужно сразу обновлять `WORK_LOG.md` и при необходимости ключевые опорные документы.
- `TASKS.md`, `docs/PROJECT_STATE.md` и `docs/DECISIONS.md` должны быть синхронизированы с крупными изменениями продукта.
- В `WORK_LOG.md` пользователь ожидает записи по смыслу изменений в проекте, а не отчет о действиях ассистента; сначала формулируется результат для продукта или процесса, потом отдельно проверки и выкладка.

## Техническая память

- Все текстовые файлы должны быть в UTF-8 без BOM.
- Нельзя доверять стандартным способам записи старого PowerShell для русских файлов.
- Основной shell для работы с текстовыми файлами и кириллицей — `pwsh` (PowerShell 7), а не `Windows PowerShell 5.1`.
- Для кириллических файлов нельзя делать массовую перезапись через старый PowerShell; безопасный путь в репозитории — `apply_patch`, `pwsh` и явный UTF-8 без BOM.
- После заметных текстовых правок нужно запускать `python tools/check_encoding.py --root .`.
- Mojibake в этом проекте часто означает проблему записи файла, а не смысловую ошибку в тексте.
- Frontend deploy flow: локальная сборка и выкладка `frontend/dist`.
- Backend deploy flow: `git pull` на хосте и перезапуск через Passenger.
- Если production `git pull` блокируется грязным рабочим деревом на сервере, нельзя слепо продолжать деплой поверх этого состояния: сначала нужно понять, что именно изменено, затем либо безопасно синхронизировать checkout с `origin/main`, либо точечно применить нужные runtime-файлы и отдельно дочистить серверный репозиторий.
- Если `apply_patch` не применился, это сигнал сначала перечитать актуальный фрагмент файла, а не пытаться силой повторять старый патч.
- Для git-операций в этом репозитории возможны `index.lock` и sandbox permission issues; безопасный путь — staged only по явному списку файлов, без массового добавления рабочего дерева.
- Архив frontend для production нельзя загружать и распаковывать параллельно: сначала upload, потом extract, потом restart.
- Если в UI появилась битая кириллица или поехал фронтенд, правильный первый шаг — не новое редактирование поверх поломки, а возврат к последнему чистому коммиту/файлу, затем `check_encoding`, затем сборка.
- Для русских строк в frontend и markdown нужно считать кодировку зоной повышенного риска: любые текстовые правки подтверждать не только чтением файла, но и проверкой сборкой/рендером.

## Память по предотвращению багов

- Если баг связан с семейной агрегацией, нужно проверять, не протекает ли кэш между per-user БД.
- Если баг приходит с мобильной версии, нужно смотреть реальную узкую раскладку, а не надеяться, что desktop-стили сами масштабируются правильно.
- Если меняется auth-flow, нужно проверять сразу три слоя:
  - backend API;
  - frontend-сообщения;
  - письма и ссылки.
- Если фикс меняет пользовательское поведение, новое правило нужно зафиксировать в документации.
- Если во время работы произошла операционная ошибка самого процесса, а не продукта, её тоже нужно переводить в проектную память как антиошибку, чтобы следующий запуск шёл по улучшенному протоколу.
- Если баг приходит из семейного режима, нужно перепроверять данные на живой семье по обоим участникам и сравнивать результат с точки зрения каждого пользователя: часть ошибок живёт не в транзакциях, а в viewer-dependent агрегации семейного dashboard/forecast.
- Если старый PR приходит из грязной или сильно отставшей ветки, не нужно мержить его целиком ради одной полезной идеи: безопаснее вытащить узкий проверенный diff в свежую ветку от текущего `main`, а старый хвост закрыть или оставить как историческую ссылку.
- Если задача про визуальную полировку, нельзя одновременно менять смысл карточек, анимацию и текстовые формулировки без промежуточной проверки: сначала вернуть чистое состояние, потом вносить одно логическое улучшение за раз.
- Если рядом уже есть блок с теми же доходами и расходами, карточка прогноза или остатка должна показывать только свой уникальный смысл: сумму остатка, тип (`личный`/`семейный`) и период прогноза.
- Для семейных категорий нельзя опираться только на сырые названия из личных БД: нужен отдельный безопасный слой сопоставления `личная категория -> семейная категория`, чтобы не ломать личную историю, но считать семейные бюджеты и forecast по единому словарю. Автосвязывание допустимо только для точных совпадений; похожие названия должны предлагаться к ручному подтверждению.

## Коммуникационная память

- Пользователь ценит непрерывность и ожидает, что ассистент помнит договоренности через документацию проекта.
- Объяснения должны быть простыми и конкретными.
- Если исправление было неправильным, это нужно говорить прямо, откатывать и чинить реальную причину.
- Для этого проекта у пользователя есть рабочее ожидание: если не сказано обратное, готовые изменения нужно не оставлять локально, а доводить до выкладки на сайт.
