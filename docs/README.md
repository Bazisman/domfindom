# Документация проекта

> Обновлено: 2026-04-24

Эта папка теперь разделена на две части:

- живые документы, по которым ведется текущая работа;
- архив, где лежат старые планы, промежуточные протоколы и исторические материалы.

## Что читать в первую очередь

1. `TASKS.md`
2. `docs/PROJECT_STATE.md`
3. `docs/DECISIONS.md`
4. `docs/WORKING_MEMORY.md`
5. `docs/WORK_LOG.md`
6. `docs/PROJECT_HISTORY.md`

## Живые документы

- `PROJECT_STATE.md` — текущее состояние продукта, архитектуры и production.
- `PROJECT_HISTORY.md` — краткая история развития проекта по этапам.
- `DECISIONS.md` — принятые продуктовые и технические договоренности.
- `FUNCTIONAL_SPEC.md` — единое описание пользовательского функционала, экранов и правил расчета.
- `GLOSSARY.md` — единый словарь терминов продукта.
- `BUSINESS_RULES.md` — формулы, расчетные правила и продуктовые инварианты.
- `CATEGORY_SYNC_SPEC.md` — ТЗ по смысловым категориям, аудиту, слиянию и семейной синхронизации.
- `SCREEN_MAP.md` — карта экранов, навигации и основных пользовательских сценариев.
- `FEATURES.md` — пользовательские возможности текущей web-версии.
- `DEPLOYMENT_GUIDE.md` — актуальная схема выкладки на `domfindom.ru`.
- `ENCODING_POLICY.md` — правила защиты от битой кодировки.
- `SETUP.md` — актуальный локальный запуск и окружение.
- `WORK_LOG.md` — основной живой протокол работ.
- `WORKING_MEMORY.md` — накопленная рабочая память проекта.
- `AI_GUIDE.md` — памятка для ИИ по текущему устройству проекта.
- `INSTRUCTIONS.md` — рабочие правила для ИИ внутри репозитория.

## Архив

Старые документы перенесены в `docs/archive/`.

Там лежат:
- старые планы миграции и внедрения;
- подробные session-файлы;
- legacy-документы desktop-эпохи;
- промежуточные протоколы, которые больше не нужны как ежедневный рабочий слой.

Важно:
- архив не удаляет историю;
- перед возвратом к старым решениям можно поднимать материалы из архива;
- новые ежедневные протоколы в архив больше не складываются по умолчанию.

## Правило протоколирования

Если пользователь пишет `запротоколируй`, в первую очередь обновляется `docs/WORK_LOG.md`.

Отдельный файл в архивном стиле нужен только если:
- произошел большой инцидент;
- нужен развернутый технический postmortem;
- нужно сохранить большую самостоятельную веху отдельно от общего журнала.
