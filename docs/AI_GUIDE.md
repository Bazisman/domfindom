## 2. Архитектура (3 слоя)

```
┌─────────────────────────────────────┐
│  GUI (views/)                       │  ← Пользователь видит это
├─────────────────────────────────────┤
│  Services (services/)               │  ← Бизнес-логика
├─────────────────────────────────────┤
│  Core (core.py)                     │  ← Работа с БД
├─────────────────────────────────────┤
│  Database (SQLite)                  │  ← Хранение данных
└─────────────────────────────────────┘
```

### Паттерны

**Observer (Слушатель):**
```python
# Вкладка подписывается на изменения
self.transaction_service.add_listener(self.on_data_changed)

# При изменении данных — вкладка обновляется
def on_data_changed(self):
    self.refresh()
```

**Сервисный слой:**
- `TransactionService` — транзакции, регулярные платежи, бюджеты
- `CategoryService` — категории

**DAO:**
- `core.py` — все SQL-запросы

**Регулярные платежи (шаблоны):**
- Транзакции с `status='planned'` — запланированные
- Транзакции с `status='actual'` — фактические
- Чтобы избежать дублирования, все запросы должны фильтровать по `status='actual'` или `status IS NULL`

### Работа с БД
```python
# Чтение
def get_items():
    with get_connection() as conn:
        cursor.execute("SELECT * FROM table WHERE is_active = 1")
        return cursor.fetchall()

# Запись (безопасная)
def update_item(item_id, **kwargs):
    allowed = ['name', 'balance']  # БЕЗОПАСНЫЕ ПОЛЯ
    with get_connection() as conn:
        cursor = conn.cursor()
        for field, value in kwargs.items():
            if field in allowed:
                cursor.execute(f"UPDATE table SET {field} = ? WHERE id = ?", (value, item_id))
        conn.commit()
```

## 5. Стиль кода

### Именование

| Тип | Пример | Правило |
|-----|--------|---------|
| Функции/методы | `get_balance()` | snake_case |
| Классы | `TransactionView` | PascalCase |
| Константы | `MAX_ITEMS = 500` | UPPER_SNAKE |
| Приватные | `_internal_method()` | _prefix |

### Типичные ошибки и решения

| Ошибка | Причина | Решение |
|--------|---------|---------|
| `invalid color name "#gray90"` | Лишние `#` в цветах CTk | Используй `("gray10", "gray90")` |
| Вкладка "В разработке" | Исключение при создании | Проверь импорты, добавь traceback |
| Данные не сохраняются | Нет вызова метода core | Проверь что вызывается метод БД |
| Окно не помещается | Маленький размер | Увеличь geometry |

### Inline-форма добавления (как в капитале)

# Фрейм с полями

## 6. Алгоритм работы над задачей

### Шаг 1: Понять задачу
- Что нужно сделать
- Где это находится
- Как должно работать
- Похожий рабочий пример в проекте

### Шаг 2: Найти контекст
- grep_search существующего кода
- Прочитать похожие реализации
- Понять паттерн

### Шаг 3: Написать код
- Следовать CODE_STYLE.md
- Использовать типичные паттерны

### Шаг 4: Проверить
- Проверить синтаксис: `python -c "from X import Y"`
- Запустить приложение
- Проверить логи при ошибках

### Шаг 5: Документировать
- Обновить CHANGELOG.md если нужно
- Записать ошибку в ERROR_LEARNING.md если была

### Шаг 6: Закрыть сессию по маркеру

Если пользователь пишет фразу:

`конец работы`

это означает обязательное завершение сессии с обновлением документации.

Минимальный чек-лист закрытия:
- обновить `TASKS.md`, если изменился статус задачи
- обновить `docs/PROJECT_STATE.md`, если изменилось текущее состояние или фокус
- обновить `docs/DECISIONS.md`, если была принята новая договорённость по логике
- обновить `docs/ERROR_LEARNING.md`, если найден новый тип ошибки или анти-паттерн
- обновить текущий файл в `docs/sessions/`, зафиксировав:
  - что проверили
  - что изменили
  - что осталось
  - следующий шаг
- обновить `docs/CHANGELOG.md`, если изменилось пользовательское поведение или функциональность

Сессию нельзя считать аккуратно завершённой, если маркер `конец работы` получен, а документация не обновлена.

### Шаг 7: Промежуточно сохранить контекст по маркеру

Если пользователь пишет фразу:

`запротоколируй`

это означает промежуточное сохранение контекста.

Минимальные действия:
- обновить `TASKS.md`
- обновить `docs/PROJECT_STATE.md`
- обновить `docs/DECISIONS.md`, если было новое правило
- обновить текущий файл в `docs/sessions/`
- при новой ошибке обновить `docs/ERROR_LEARNING.md`

Отличие от `конец работы`:
- `запротоколируй` не завершает работу окончательно
- после него можно продолжать ту же сессию дальше

## 7. Работа с документацией

| Файл | Назначение |
|------|------------|
| `docs/README.md` | Главная страница, как запустить |
| `docs/PROJECT_STATE.md` | Текущее состояние проекта и текущий фокус |
| `docs/DECISIONS.md` | Зафиксированные договорённости по логике |
| `docs/ARCHITECTURE.md` | Архитектура и БД |
| `docs/API.md` | Основные методы |
| `docs/CODE_STYLE.md` | Правила стиля кода |
| `docs/CHANGELOG.md` | История версий |
| `docs/ERROR_LEARNING.md` | Грабли и как их избежать |

### Порядок чтения в новой сессии

Если ИИ попал в новый чат или модель не знает контекст, читать в таком порядке:

1. `KODA.md`
2. `docs/PROJECT_STATE.md`
3. `docs/DECISIONS.md`
4. последнюю запись в `docs/sessions/`
5. `docs/ERROR_LEARNING.md`

Это обязательный минимум перед серьёзными правками.

### Какие документы обновлять после работы

- `TASKS.md` — когда изменился статус задачи или следующий шаг
- `docs/PROJECT_STATE.md` — когда изменилось текущее состояние проекта
- `docs/DECISIONS.md` — когда принято новое правило логики
- `docs/ERROR_LEARNING.md` — когда обнаружена новая ошибка, грабли или повторяемый анти-паттерн
- `docs/sessions/*.md` — в конце каждой содержательной рабочей сессии
- `docs/CHANGELOG.md` — только для заметных функциональных изменений

## 8. Частые грабли

1. **Изменения не применяются** → Перезапусти приложение
2. **Ошибка импорта** → Проверь: `python -c "from views.XXX import YYY"`
3. **Окно не помещается** → Увеличь geometry или сделай resizable
4. **Данные не сохраняются** → Проверь вызов метода core.py

## 9. Как учиться на ошибках

После каждой ошибки:
1. Запиши что произошло
2. Проанализируй причину
3. Запиши в `docs/ERROR_LEARNING.md`
4. В следующий раз вспомни и не повторяй

**Пример записи:**
```markdown
### 2026-03-30: invalid color name "#gray90"

**Проблема:** Лишние # в цветах для CustomTkinter

**Причина:** Копировал код из другого места где были hex-цвета

**Решение:** Использовать ("gray10", "gray90") без #

**Как избежать:** Запомнить — CTk использует имена цветов, не hex
```
